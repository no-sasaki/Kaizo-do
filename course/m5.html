① 依頼を受ける

M4 のプレスルームが好評で、社内の信頼が一段上がりました。今度はデジタルマーケ責任者の三宅さんから、ジャーナル（編集記事）についての相談です。

💬 デジタル・三宅 亮 からの依頼（R-02）

「ジャーナル、記事は増えてきたんですけど、読者が目的の記事にたどり着けてないんですよね。 “成分”とか“効能”で探せるようにできませんか？ 『保湿の記事だけ読みたい』『発酵の話を全部読みたい』みたいに。回遊とオーガニック流入を伸ばしたいんです。」

M4 で category を型に入れ忘れると絞り込みが作れなかった ── あの感覚を思い出してください。今回はそれがもう一段大きなスケールで効きます。「成分・効能で探せる」は、ボタンの話ではなく 記事の型に、何を・どんな構造で持たせるかの話だからです。

② AIと壁打ちして要件を定義する

ここでも、握るべきを握ってから委ねます。今回いちばん握るべきは ──「“成分・効能”を、自由記述で持つのか、統制された語彙（タグ）で持つのか」という一点です。ここが情報設計の分かれ目になります。

▷ AIに委ねる

  • ユーザーストーリー・受け入れ条件の草案
  • 記事に必要なフィールド（型）の候補出し
  • 成分・効能の統制語彙（タグ一覧）のたたき台
  • ファセット（絞り込み軸）UIのパターン

● 人が握る

  • 自由記述ではなく統制語彙にする判断
  • 成功の定義＝「目的の記事に3クリック以内で到達」
  • タグの粒度（細かすぎ／粗すぎを避ける）
  • 記事↔成分↔製品をつなぐ設計判断

握ったら、書く ── ここで決めた判断は、コンテンツモデルと統制語彙（taxonomies）表に書き留めて初めて「握った」ことになります。

まず“発散”。型と語彙の論点を、AIに洗い出させます。

🧭 コンサルの思考発散：型と語彙の論点
あなたは経験豊富なWCM（Webコンテンツ管理）／情報設計のコンサルタントです。
ブランドサイト「NAGI」の編集記事（ジャーナル）を「成分・効能で探せる」ようにします。
記事は構造化データ(JSON)で管理され、表示はクライアントで描画されます。

私が判断するための“発散”をしてください。
1. 「成分・効能で探せる」を成立させるために、記事の型(コンテンツモデル)に必要なフィールドは？
2. 成分・効能を「本文中の自由記述」で持つ場合と「統制語彙(タグID＋表示ラベル)」で持つ場合の、
   絞り込み・多言語・表記ゆれ・運用・拡張性の観点での違いを表で。
3. 統制語彙にする場合の、タグIDの設計指針（粒度・命名・i18nラベルの持ち方）。
4. 記事↔成分↔製品を“つなぐ”ことで広がる価値（回遊・関連表示・SEO）。
5. 見落としがちなエッジケース（タグ0件の記事、未翻訳ラベル、複数タグ、廃止タグ）。
最後に、私が必ず自分で決めるべき論点を3つ指摘してください。

AIは「自由記述は手軽だが表記ゆれで壊れる」「統制語彙は設計が要るが発見性・多言語・関係性に強い」といった対比を返します。ここで握るのは、その対比を三宅さんの成功の定義（回遊・流入）に照らして判断すること。NAGIは多言語・多地域で、記事↔製品の回遊を伸ばしたい ── ならば答えは統制語彙です。

次に“収束”。決めた方針を、検証可能な成果物に落とさせます。

🧭 コンサルの思考収束：モデルと受け入れ条件
方針を確定します。
- 成分・効能は「統制語彙(タグ)」で持つ。タグはID(不変)＋表示ラベル(ja/en)を taxonomies に集約。
- 成功の定義：読者が目的の記事に3クリック以内で到達でき、関連製品にも回遊できる。
- スコープに入れる：成分での絞り込み、効能での絞り込み、複数選択、0件表示、ja/enラベル、記事↔製品の関連表示。
- 今回は外す：全文検索、タグの自動推定、タグ管理画面。

これを、実装とテストができる「ユーザーストーリー＋受け入れ条件(Given/When/Then)」に落とし、MoSCoWで優先度を付けてください。
あわせて、記事の型(コンテンツモデル)の追加フィールド一覧（名前／型／必須・任意／多言語の有無／なぜ必要か＝どの受け入れ条件を支えるか）と、
成分・効能の統制語彙(タクソノミー)の初版（タグID と ja/en ラベル）を表で出してください。

● 出力は一例 AIの結果は実行ごとに変わります。大事なのは、返ってきた草案を成功の定義と照らして取捨すること ── それが「握る」です。プロンプトを自分で書けるようになるには →（Appendix）

ナビゲーション方針（論理サイトマップ抜粋）IA
  • ジャーナル一覧の上部にファセット（成分／効能）を置く。グローバルナビは据え置き（増やさない）。
  • 選択状態はURLに持つ（例 ?ingredient=rice-ferment）。共有・ブックマーク・SEO・計測のため。
  • 記事詳細の末尾に関連製品（同タグ）を表示し、回遊を作る。
  • タグのラベルは多言語、IDは不変。市場が増えてもURL・データは壊れない。

なぜ「統制語彙」にこだわるのか ── 型の質は、発見性の質になる

③で決めた「自由記述ではなく統制語彙」という判断 ── これが発見性をどれだけ左右するかを、実装に入る前に見ておきましょう。同じ「成分で探す」でも、持ち方が違うだけで結果が割れます。

● 以下は一例 見るべきは「壊れ方の傾向」です（このあと自分でも触れます）。

✗ 自由記述で持つと

記事の持ち方：成分を本文や自由テキストで記述。

A: "米発酵液"
B: "発酵液"
C: "ライス発酵エキス"
  • 🔴 同じ成分が3つの別物に分裂（表記ゆれ）
  • 🔴 「発酵で絞る」と一部の記事が取りこぼれる
  • 🔴 効能の軸が無く、保湿で探せない
  • 🔴 ラベルの置き場が無く多言語化できない
  • 🔴 製品の成分と突き合わせられず回遊が作れない

発見性が壊れる。UIをいくら作り込んでも、土台の語彙がばらつくと“探せない”。

✓ 統制語彙（タグ）で持つと

記事の持ち方：タグID＋taxonomies のラベル。

A: ["rice-ferment"]
B: ["rice-ferment"]
C: ["rice-ferment"]
taxonomies.rice-ferment
  = {ja:"米発酵液", en:"Rice Ferment"}
  • 🟢 同じ成分は1つのファセットに統一
  • 🟢 取りこぼしなく絞り込める
  • 🟢 効能タグも同じ仕組みで追加できる
  • 🟢 ラベルだけ差し替えてja/en対応
  • 🟢 製品の同タグと関連表示が作れる

発見性が育つ。差を生んだのはUIではなく、上流の語彙の設計。

自分で触って確かめる ── インタラクティブ

左の記事データの ingredients を書き換えると、右のファセット絞り込みがその場で再描画されます。 「統制語彙（タグ）」と「自由記述（表記ゆれ）」のプリセットを切り替えて、同じ「米発酵液」の記事がいくつのファセットに分裂するかを見比べてください。English に切り替えると、自由記述ではラベルが崩れます。

    ✦ ここに、レバレッジが見える

    「米発酵液」「発酵液」「ライス発酵」── 人が書けば必ずゆれる表記が、絞り込みを静かに壊します。上流でタグIDを一つに統制するという小さな判断が、下流の発見性・多言語・回遊をまとめて決めている。情報設計とは、このことばの統制のことです。

    ④ AIに実装させる ── 自分のNAGIサイトに反映する

    “統制語彙の方”を、自分の砂場（Fork した NAGI）に実装します。ブラウザの GitHub Web エディタとチャットAIだけで完結します。

    4-1. コンテンツ（型のインスタンス）は、もう用意されている

    ジャーナルのデータは、すでにタグ付きで用意されています ── nagi/data/journal.json を見る ↗。 各記事に ingredients / benefits（タグID）と related_products が入り、表示ラベルは taxonomies に ja/en で集約されています。中身は触らず、これを「探せる表示層」にします。

    📦 型を見れば、設計が読める

    journal.json の taxonomies と各記事の ingredients を見比べると、③で設計した「ID＋ラベル分離」がそのまま実データになっているのが分かります。設計＝データ＝実装が一直線につながります。

    4-2. 表示層（ファセット絞り込み）をAIに作らせる

    表示層のコードを覚える必要はありません。仕様（受け入れ条件）と既存の作法をAIに渡し、生成させます。

    このプロンプトは実装体験用です。自分で書けなくて問題ありません。見るのは 保存先ファイル・満たす条件・貼り付け位置の3つだけ。

    ⚙️ 開発体験用・コピペでOK実装：ジャーナルに成分・効能の絞り込みを追加
    あなたはフロントエンド実装者です。既存の静的サイト「NAGI」のジャーナルに、ファセット絞り込みを追加します。
このサイトは各ページが /data/*.json を fetch してDOMに描画する「擬似ヘッドレス」構成です。
（参考：journal.html は data/journal.json を読み、一覧と ?id= の詳細を描画している。）

データ仕様：journal.json は items[] と taxonomies を持つ。
- 各記事は ingredients[]・benefits[]（taxonomies のタグID）、related_products[] を持つ。
- taxonomies.ingredient / .benefit は タグID → { ja, en } ラベル。

journal.html の一覧に、次を満たすファセット絞り込みを追加してください：
- 一覧の上部に「成分」「効能」のタグ群を、件数つきで表示（taxonomies のラベルを使用）。
- タグをクリックで絞り込み、成分×効能は複数選択でAND。選択状態は URL クエリに持つ。
- 0件のときは一覧領域に「該当する記事はありません」と絞り込み解除リンクを表示すること。
- 言語切替でタグの表示ラベルが ja/en で切り替わる（タグIDは不変）。ingredients/benefits が空配列の記事も一覧に表示し、ファセットの件数計算では対象外として扱うこと。
- 既存の learner-banner / nav / footer / デザイントークン(assets/css/nagi.css)を踏襲。

変更後の journal.html の完全なコードと、必要なら最小の追加JSを出力してください。

    気づき 1

    表記ゆれは、静かに壊す

    「発酵液」と「米発酵液」を別物にした瞬間、絞り込みは取りこぼす。エラーは出ない。だから設計図上は気づけない。

    気づき 2

    同じタグが、回遊を生む

    記事と製品が同じ統制タグを共有するから、関連表示が自動で成立する。情報設計は、KPI（回遊・流入）に直結する。

    十分性チェック

    セルフチェック合意の前に
    • □ 成分・効能を統制語彙（タグID＋ラベル分離）で定義したか
    • □ 各フィールドが受け入れ条件に紐づいているか（なぜ必要かが言えるか）
    • □ エッジケース（0件・タグ無し記事・未翻訳ラベル・複数選択）が条件にあるか
    • □ 記事↔成分↔製品の関係と、ファセットの置き場（ナビ）を決めたか
    • □ 成功の定義（3クリックで到達・回遊）に照らして過不足がないか

    ✦ となりのUX要件（M3で扱った視点）

    ファセットUIは UX 要件の宝庫です ── 選択中タグの明示、件数表示、「すべて解除」、0件のときの導線、複数選択の操作感。これらは状態と振る舞いで、実装して触って初めて良し悪しが分かります。M3で扱った「状態と振る舞い」の観点が、このファセットUIにもそのまま効きます。

    次へ

    型を「点」から「つながり」へ広げ、ことばの統制が発見性を決めることを体感しました。要件定義の解像度に、いまや「タクソノミー」「ファセット」「関係」「IDとラベルの分離」が自然に入っているはずです。
