① 依頼を受ける

あなたのSlackに、広報の林さんからメッセージが届きました。

💬 PR・林 美咲 からの依頼

「お疲れさまです。来月、海外向けに大きな発表があって、記者にプレスリリースをちゃんと見せられるページがサイトに欲しいんです。今はジャーナルに混ぜて載せてるんですけど、報道の人が探しにくいみたいで……。いい感じにお願いできますか？」

典型的な、現実の依頼です。悪気なく曖昧で、判断材料が足りない。ここで多くの人がやってしまうのが ──「プレスリリース一覧ページを作る」とだけ理解して、すぐ手を動かすこと。この理解の粗さが、後で実装を壊します。

まず、依頼の裏にある問いを立てます。誰が・何のために・どんな状況で使う？ 既存のジャーナルと何が違う？ 海外向けとは？ 「探しにくい」の正体は？

② AIと壁打ちして要件を定義する

ここからAIの出番です。ただし ── 丸投げではなく、握るべきを握ったうえで委ねる。

▷ AIに委ねる

  • ユーザーストーリーの候補出し
  • 受け入れ条件のドラフト
  • エッジケースの洗い出し（多言語・日付・空状態…）
  • 必要なコンテンツ項目（フィールド）の提案

● 人が握る

  • 成功の定義＝「記者が3分で目的の資料に到達できる」
  • ブランド制約（NAGIの静謐な世界観を壊さない）
  • MoSCoW の優先度づけ
  • 「これで関係者が揉めないか」の判定

まずは“発散”。AIにペルソナと利用文脈から考えさせ、抜けを潰します。下のプロンプトをコピーして、あなたのチャットAIに貼ってください。

Prompt ─ 発散：要件の洗い出し
あなたは経験豊富なWCM（Webコンテンツ管理）のコンサルタントです。
京都発の架空ウェルネスブランド「NAGI」のブランドサイト（静的・JSON駆動）に、
新しいコンテンツタイプ「プレスリリース」を追加します。

依頼（広報担当の生の言葉）:
「海外の記者にプレスリリースをちゃんと見せたい。今はジャーナルに混ざっていて探しにくい。」

次を出してください。私が判断するための“発散”が目的です。
1. 主要な利用者（ペルソナ）と利用文脈を3つ。各ペルソナの「成功＝何ができれば満足か」も。
2. このページで満たすべきユーザーストーリーの候補を5〜8個（As a / I want / so that 形式）。
3. プレスリリースが「ジャーナル記事」と決定的に異なる点（コンテンツ項目の違い）を表で。
4. 見落としがちなエッジケース（多言語、公開日、報道解禁、PDF、空状態、過去アーカイブ等）を列挙。
最後に、私が必ず自分で決めるべき論点を3つ指摘してください。

AIは多くの候補を返します。ここであなたが握るのは取捨選択です。「報道解禁日」は要る？「カテゴリ（製品/企業/サステナ）」は？「言語フィルタ」は最初から必要か、後回しか？ ── 成功の定義（記者が3分で目的に到達）に照らして、入れる/落とすを決めます。

次に“収束”。決めた方針をAIに渡し、合意可能な形式に整えさせます。

Prompt ─ 収束：受け入れ条件への落とし込み
先の検討から、以下の方針で確定します。
- 成功の定義: 海外の記者が、サイトに来てから3分以内に「最新の/目的の」プレスリリース本文とメディア向け資料に到達できる。
- スコープに入れる: 一覧（新しい順）、カテゴリでの絞り込み（製品/企業/サステナ）、公開日、本文、報道用の問い合わせ先、日英の出し分け。
- 今回は外す: 全文検索、PDF自動生成、購読通知。

これを、実装とテストができる「ユーザーストーリー＋受け入れ条件（Given/When/Then）」に落としてください。
各ストーリーに MoSCoW（Must/Should/Could/Won't）の優先度を付けること。
受け入れ条件は曖昧語（"いい感じに"等）を排し、検証可能な記述にすること。
さらに、この機能を支えるのに必要な「プレスリリースのコンテンツモデル（フィールド一覧／型／必須・任意／多言語対応の有無）」を表で添えてください。

✦ ここがAI活用の本質

AIは「量」と「網羅」を一瞬で出す。人は「成功の定義」という軸を与え、その軸で切る。軸がなければAIの量はノイズになり、軸があればAIの量は抜け漏れのない要件になる。技術が見えていると、この軸に「実装で効く観点（型・日付・多言語）」が自然と入る ── それが青写真しか描けない状態との差です。

③ 成果物をつくる

AIの出力を、あなたの判断で整えた“合意できる成果物”がこれです。ファイル形式は問いません ── 重要なのは本質的な観点が網羅され、検証可能であること。

成果物A ｜ ユーザーストーリー＋受け入れ条件＋優先度T9 OUT
IDユーザーストーリー受け入れ条件（Given/When/Then 抜粋）優先度
US-1 海外の記者として、最新のプレスリリースを一覧で見たい。最新の発表を逃さないため。 プレスルームを開くと、リリースが公開日の新しい順で並ぶ。各項目に日付・カテゴリ・見出しが表示される。 Must
US-2 記者として、関心領域（製品/企業/サステナ）で絞り込みたい。目的の発表に早く着くため。 カテゴリを選ぶと、そのカテゴリのリリースだけが残る。該当0件のとき「該当なし」を表示し、壊れない。 Must
US-3 記者として、本文と報道用の問い合わせ先を1ページで読みたい。 個別ページに、本文・公開日・カテゴリ・広報連絡先が表示される。 Must
US-4 海外の記者として、英語のリリースを読みたい。日本語が読めないため。 言語切替で、見出し・本文が当該言語で表示される。未翻訳項目はフォールバック表示し、欠落で崩れない。 Should
US-5 記者として、過去のリリースも辿りたい。 年でアーカイブを辿れる。 Could
US-6 （全文検索） ── Won't（今回）

受け入れ条件US-2の「該当0件のとき壊れない」、US-4の「欠落で崩れない」── こうしたエッジケースを“先に”言語化できるかが、技術が見えている要件定義の証拠です。これが抜けると、後で実装が崩れます（次節で実演します）。

成果物B ｜ コンテンツモデルの種（プレスリリース型）T10 連動
フィールド型必須多言語なぜ必要か（どの受け入れ条件を支えるか）
id文字列（一意）必須—個別ページのURL・参照（US-3）
category列挙（product/corporate/sustainability）必須—絞り込み（US-2） ← これが無いとフィルタが作れない
date日付 ISO必須—新しい順の並び替え（US-1）・アーカイブ（US-5）
titleテキスト必須有見出し（US-1, US-4）
bodyリッチ/段落配列必須有本文（US-3, US-4）
media_contactオブジェクト（氏名/email）必須—報道連絡先（US-3）
embargo真偽/日時任意—報道解禁の制御（将来）

「なぜ必要か」の列に注目。各フィールドが特定の受け入れ条件に紐づいています。これがトレーサビリティ（要求↔要件の追跡）の芽です。M9/M10で本格的に扱いますが、種はここで蒔かれます。

⚡ ここで、わざと一度失敗します

主張を本当に信じるには、浅いやり方の失敗を一度、自分で見るのが最短です。同じ依頼を、2つの要件でAIに実装させ、結果を比べます。

✗ 浅い要件で実装させると

渡した指示：「プレスリリース一覧ページを作って。ジャーナルみたいな感じで。」

渡したデータ（型が雑）：

{ "title": "新エッセンス発売",
  "text": "...本文..." }

AIの出力：それっぽい一覧。だが ──

  • 🔴 カテゴリで絞り込めない（型に category が無い）
  • 🔴 新しい順に並ばない（date が無い／文字列）
  • 🔴 英語版を出そうとすると多言語の置き場所が無く崩れる
  • 🔴 報道連絡先が無く、記者が問い合わせ先に辿れない
  • 🔴 0件時の表示が未定義で空白orエラー

手戻り発生。型を直す＝データもコードも作り直し。これが「実装フェーズで初めて表面化する盲点」の正体。

✓ 深い要件で実装させると

渡した指示：成果物A（受け入れ条件）＋ 成果物B（コンテンツモデル）＋ 構造化データ。

渡したデータ（型が効いている）：

{ "id":"essence-launch",
  "category":"product",
  "date":"2026-06-01",
  "title":{"ja":"…","en":"…"},
  "media_contact":{…} }

AIの出力：受け入れ条件を満たす一覧。

  • 🟢 カテゴリのフィルタがそのまま作れる
  • 🟢 日付で正しく新しい順に並ぶ
  • 🟢 言語切替が破綻しない（フォールバック付き）
  • 🟢 連絡先・0件表示まで定義済み

一発で通る。滑らかで、品質が高い。差を生んだのはAIの賢さではなく、あなたの要件の質。

🔥 食らってほしい瞬間

「カテゴリで絞り込めると思っていたのに、型に category を入れ忘れていたから絞り込みが作れない」── この壁は、設計図の上では決して見えません。手を動かして、自分のサイトが壊れて、初めて見える。それを一度経験すると、次からあなたの要件定義には自然と「型」「日付」「多言語」「空状態」が入るようになります。

4-2. 表示層（press.html）をAIに作らせる

あなたは表示層のコードを覚える必要はありません。仕様（受け入れ条件）と既存の作法をAIに渡し、生成させます。下のプロンプトを使ってください。

Prompt ─ 実装：press.html を生成
あなたはフロントエンド実装者です。既存の静的サイト「NAGI」に、プレスルーム機能を追加します。
このサイトは、各ページが /data/*.json を fetch してDOMに描画する「擬似ヘッドレス」構成です。
（参考：products.html は data/products.json を、journal.html は data/journal.json を読み、?id= で詳細表示に切り替わる。）

作るもの: nagi/press.html （一覧＋?id=での詳細）。同じ作法・同じデザイントークン(assets/css/nagi.css)を使うこと。

満たす受け入れ条件:
- 一覧は data/press.json を読み、date の新しい順に並べる。各項目に date・category_label・title を表示。
- カテゴリ(product/corporate/sustainability)で絞り込むボタンを置く。0件のときは「該当するリリースはありません」と表示し、壊れないこと。
- ?id= のとき、その詳細（title, date, category, body, media_contact）を表示する。
- title/summary/body は { "ja":..., "en":... } の多言語。表示言語が無ければ ja にフォールバックし、欠落で崩れないこと。
- 既存の learner-banner / nav / footer の構造を踏襲すること。

press.html の完全なHTMLと、必要ならpress用の最小JSを出力してください。

生成されたコードを GitHub Web エディタで nagi/press.html として保存。さらに NAGI のナビに「Press」リンクを足せば、記者が辿れます。コミットすると、数十秒〜数分であなたの GitHub Pages に反映されます。

🧪 動いたら、受け入れ条件で“検収”する

実装が終わったら、開発者ではなく発注者の目で確かめます。成果物Aの受け入れ条件を一行ずつ指差し確認：新しい順に並ぶ？ カテゴリで絞れる？ 0件で壊れない？ 英語に切り替わる？ ── これが「Done の共有」。要件があるから、検収が“できる”のです。

⑤ 結果を体験し、振り返る

動いたプレスルームを眺めてください。そして、いちばん大事な作業 ── 言語化をします。

②（要件定義）の質が、⑤（実装の品質）を決めた。
AIは賢くなった。だが、AIに渡す要件の解像度が、そのまま成果物の解像度になる。技術が見えていると、その解像度に「型・日付・多言語・空状態」が最初から含まれる。
気づき 1

盲点は、実装で初めて痛い

「カテゴリ」を型に入れ忘れる程度のことが、絞り込み機能そのものを不可能にした。上流の小さな抜けが、下流で大きな手戻りになる。

気づき 2

AIは“握り”があって増幅する

成功の定義という軸を握ってから委ねると、AIの「量」が「網羅」に変わる。軸が無ければ、同じAIが速くノイズを出す。

十分性チェック ──「揉めない＝検証可能」

合格ラインは「設計フェーズの担当者が、解釈の余地で手が止まらない」状態です。次に自信を持って“はい”と言えますか。

セルフチェック合意の前に
  • □ 各ストーリーに、検証可能な受け入れ条件があるか（"いい感じ"が残っていないか）
  • □ 優先度（MoSCoW）が付き、今回やらないこと（Won't）が明示されているか
  • □ エッジケース（0件・未翻訳・並び順）が条件に含まれているか
  • □ 機能を支えるコンテンツモデルの種が定義され、各フィールドが受け入れ条件に紐づくか
  • □ 成功の定義（記者が3分で到達）に照らして、過不足がないか

次へ

あなたは体験ループを1周まわしました。曖昧な依頼 → 握って委ねる壁打ち → 検証可能な成果物 → 実装 → 検収と振り返り。これがすべてのモジュールの背骨です。

次の M5（情報設計＆コンテンツモデル）では、今回「種」として触れたコンテンツモデリングを、サイト全体の情報設計として深く掘ります。型の質が機能の質を決めることを、もう一段大きなスケールで体感します。
