① 依頼を受ける

Slack に、広報の林さんからメッセージが届きました。

💬 PR・林 美咲 からの依頼

「お疲れさまです。来月、海外向けに大きな発表があって、記者にプレスリリースをちゃんと見せられるページがサイトに欲しいんです。今はジャーナルに混ぜて載せてるんですけど、報道の人が探しにくいみたいで……。いい感じにお願いできますか？」

典型的な、現実の依頼です。悪気なく曖昧で、判断材料が足りない。ここで多くの人がやってしまうのが ──「プレスリリース一覧ページを作る」とだけ理解して、すぐ手を動かすこと。この理解の粗さが、後で実装を壊します。

まず、依頼の裏にある問いを立てます。誰が・何のために・どんな状況で使う？ 既存のジャーナルと何が違う？ 海外向けとは？ 「探しにくい」の正体は？

② AIと壁打ちして要件を定義する

ここからAIの出番です。ただし ── 判断軸なしに任せるのではなく、握るべきを握ったうえで委ねる。

📋 プロンプトカードの読み方

以下、コピーして使えるプロンプトが並びます。各カードには2種類のラベルが付きます ── 🧭 コンサルの思考＝自分で書けるようになりたい側、 ⚙️ 開発体験用・コピペでOK＝サイト固有の前提を含むので、いまはそのまま使って十分な側。 AIの出力は実行ごとに変わるので、教材の結果と違っても正常です。 自分で書けるようになるには →（Appendix）

▷ AIに委ねる

  • ユーザーストーリーの候補出し
  • 受け入れ条件のドラフト
  • エッジケースの洗い出し（多言語・日付・空状態…）
  • 必要なコンテンツ項目（フィールド）の提案

● 人が握る

  • 成功の定義＝「記者が3分で目的の資料に到達できる」
  • ブランド制約（NAGIの静謐な世界観を壊さない）
  • MoSCoW の優先度づけ
  • 「これで関係者が揉めないか」の判定

握ったら、書く ── ここで決めた判断は、受け入れ条件と MoSCoW の優先度に書き留めて初めて「握った」ことになります。

まずは“発散”。AIにペルソナと利用文脈から考えさせ、抜けを潰します。下のプロンプトをコピーして、チャットAIに貼ってください。

🧭 コンサルの思考発散：要件の洗い出し
あなたは経験豊富なWCM（Webコンテンツ管理）のコンサルタントです。
京都発の架空ウェルネスブランド「NAGI」のブランドサイト（静的・JSON駆動）に、
新しいコンテンツタイプ「プレスリリース」を追加します。

依頼（広報担当の生の言葉）:
「海外の記者にプレスリリースをちゃんと見せたい。今はジャーナルに混ざっていて探しにくい。」

次を出してください。私が判断するための“発散”が目的です。
1. 主要な利用者（ペルソナ）と利用文脈を3つ。各ペルソナの「成功＝何ができれば満足か」も。
2. このページで満たすべきユーザーストーリーの候補を5〜8個（As a / I want / so that 形式）。
3. プレスリリースが「ジャーナル記事」と決定的に異なる点（コンテンツ項目の違い）を表で。
4. 見落としがちなエッジケース（多言語、公開日、報道解禁、PDF、空状態、過去アーカイブ等）を列挙。
最後に、私が必ず自分で決めるべき論点を3つ指摘してください。

AIは多くの候補を返します。ここで握るのは取捨選択です。「報道解禁日」は要る？「カテゴリ（製品/企業/サステナ）」は？「言語フィルタ」は最初から必要か、後回しか？ ── 成功の定義（記者が3分で目的に到達）に照らして、入れる/落とすを決めます。

次に“収束”。決めた方針をAIに渡し、合意可能な形式に整えさせます。

🧭 コンサルの思考収束：受け入れ条件への落とし込み
先の検討から、以下の方針で確定します。
- 成功の定義: 海外の記者が、サイトに来てから3分以内に「最新の/目的の」プレスリリース本文とメディア向け資料に到達できる。
- スコープに入れる: 一覧（新しい順）、カテゴリでの絞り込み（製品/企業/サステナ）、公開日、本文、報道用の問い合わせ先、日英の出し分け。
- 今回は外す: 全文検索、PDF自動生成、購読通知。

これを、実装とテストができる「ユーザーストーリー＋受け入れ条件（Given/When/Then）」に落としてください。
各ストーリーに MoSCoW（Must/Should/Could/Won't）の優先度を付けること。
受け入れ条件は曖昧語（"いい感じに"等）を排し、検証可能な記述にすること。
さらに、この機能を支えるのに必要な「プレスリリースのコンテンツモデル（フィールド一覧／型／必須・任意／多言語対応の有無）」を表で添えてください。

✦ ここがAI活用の本質

AIは「量」と「網羅」を一瞬で出す。人は「成功の定義」という軸を与え、その軸で切る。軸がなければAIの量はノイズになり、軸があればAIの量は抜け漏れのない要件になる。技術が見えていると、この軸に「実装で効く観点（型・日付・多言語）」が自然と入ります。

③ 成果物をつくる

AIの出力を、自分の判断で整えた“合意できる成果物”がこれです。ファイル形式は問いません ── 重要なのは本質的な観点が網羅され、検証可能であること。

なぜ、ここまで成果物にこだわるのか ── 成果物の質は、実装の質になる

③でつくった成果物（受け入れ条件＋コンテンツモデル）は、このあと ④で実装に渡します。その前にひとつ ── 「成果物の質が、実装の結果をどう左右するか」を、小さな実験で確かめておきましょう。ここが腑に落ちると、なぜ要件定義にこれだけ手間をかけるのかが、体でわかります。

方法はシンプルです。同じAI・同じ依頼に、粗いブリーフと、いま整えた精緻なブリーフを渡し、出力を見比べるだけ。差を生むのはAIの賢さではなく、ブリーフの解像度です。

● 以下は一例 AIの出力は実行ごとに変わるため、手元の結果が下と違っても問題ありません。見るべきは「壊れ方の傾向」です（このあと、自分でも触って確かめられます）。

✗ 粗いブリーフで実装させると

渡した指示：「プレスリリース一覧ページを作って。ジャーナルみたいな感じで。」

渡したデータ（型が雑）：

{ "title": "新エッセンス発売",
  "text": "...本文..." }

AIの出力：それっぽい一覧。だが ──

  • 🔴 カテゴリで絞り込めない（型に category が無い）
  • 🔴 新しい順に並ばない（date が無い／文字列）
  • 🔴 英語版を出そうとすると多言語の置き場所が無く崩れる
  • 🔴 報道連絡先が無く、記者が問い合わせ先に辿れない
  • 🔴 0件時の表示が未定義で空白orエラー

手戻り発生。型を直す＝データもコードも作り直し。これが「実装フェーズで初めて表面化する見落とし」の正体。

✓ 精緻なブリーフで実装させると

渡した指示：成果物A（受け入れ条件）＋ 成果物B（コンテンツモデル）＋ 構造化データ。

渡したデータ（型が効いている）：

{ "id":"essence-launch",
  "category":"product",
  "date":"2026-06-01",
  "title":{"ja":"…","en":"…"},
  "media_contact":{…} }

AIの出力：受け入れ条件を満たす一覧。

  • 🟢 カテゴリのフィルタがそのまま作れる
  • 🟢 日付で正しく新しい順に並ぶ
  • 🟢 言語切替が破綻しない（フォールバック付き）
  • 🟢 連絡先・0件表示まで定義済み

一発で通る。滑らかで、品質が高い。差を生んだのはAIの賢さではなく、要件の質。

自分で触って確かめる ── インタラクティブ

読むだけでなく、いま手元で。左の「型（JSON）」を書き換えると、右のプレスルームがその場で再描画されます。 category を1行消すと絞り込みが消え、date を消すと並び替えが止まり、title を文字列にすると多言語が崩れる ── 上流の一行が下流の機能を決めることを、AIもForkも使わず体感できます。

    ✦ ここに、レバレッジが見える

    「カテゴリで絞り込めると思っていたのに、型に category を入れ忘れていたから絞り込みが作れない」── この壁は、設計図の上では決して見えません。手を動かし、サイトの挙動として現れて、初めて見える。これを一度通すと、要件定義には自然と「型」「日付」「多言語」「空状態」が入るようになります。上流の小さな一行が、下流をこれだけ動かす ── それがレバレッジです。

    ④ AIに実装させる ── 自分のNAGIサイトに反映する

    では“精緻な方”を、自分の砂場（Fork した NAGI）に実装します。ブラウザの GitHub Web エディタとチャットAIだけで完結します。

    4-1. コンテンツ（型のインスタンス）は、もう用意されている

    プレスリリースの中身は、すでに NAGI サイトに構造化データとして置かれています ── nagi/data/press.json を見る ↗。 これは成果物B（コンテンツモデル）の「型」に従った実データで、category・date・多言語の title・media_contact などが入っています。中身は触らなくて大丈夫。これから作るのは、このデータを画面に出す「表示層」です。

    実務でも、コンテンツ（中身）と表示（見た目）は別の担当・別の工程で進みます。ここでは中身を用意済みにして、「要件 → 表示の実装」に集中できるようにしています。中身の構造は press.json を開けばいつでも確認できます。

    🧪 動いたら、受け入れ条件で“検収”する

    実装が終わったら、開発者ではなく発注者の目で確かめます。成果物Aの受け入れ条件を一行ずつ指差し確認：新しい順に並ぶ？ カテゴリで絞れる？ 0件で“該当なし”が出る？ 英語に切り替わる？ ── これが「Done の共有」。要件があるから、検収が“できる”のです。

    ⑤ 結果を体験し、振り返る

    動いたプレスルームを眺めてください。そして、いちばん大事な作業 ── 言語化をします。

    ③（成果物）の質が、⑤（実装の品質）を決めた。
    AIは賢くなった。だが、AIに渡す要件の解像度が、そのまま成果物の解像度になる。技術が見えていると、その解像度に「型・日付・多言語・空状態」が最初から含まれる。
    気づき 1

    見落としは、実装で初めて見える

    「カテゴリ」を型に入れ忘れる程度のことが、絞り込み機能そのものを不可能にした。上流の小さな一行が、下流で大きな手戻りになる。

    気づき 2

    AIは“握り”があって増幅する

    成功の定義という軸を握ってから委ねると、AIの「量」が「網羅」に変わる。軸が無ければ、同じAIが速くノイズを出す。

    十分性チェック ──「揉めない＝検証可能」

    合格ラインは「設計フェーズの担当者が、解釈の余地で手が止まらない」状態です。次に自信を持って“はい”と言えますか。

    セルフチェック合意の前に
    • □ 各ストーリーに、検証可能な受け入れ条件があるか（"いい感じ"が残っていないか）
    • □ 優先度（MoSCoW）が付き、今回やらないこと（Won't）が明示されているか
    • □ エッジケース（0件・未翻訳・並び順）が条件に含まれているか
    • □ 機能を支えるコンテンツモデルの種が定義され、各フィールドが受け入れ条件に紐づくか
    • □ 成功の定義（記者が3分で到達）に照らして、過不足がないか

    そのとなりに、UX要件がある

    機能要件は、比較的たどり着きやすい要件です ──「何ができるか」を言葉にすればいい。一方で UX要件は言語化しにくく、実装を見たことがないと書けないことが多い。でも実は、その入り口はもうこのモジュールの中にありました。

    ✦ 「0件で“該当なし”を出す」は、実はUX要件

    受け入れ条件に入れた「該当0件のとき“該当なし”を出す」「未翻訳は ja にフォールバックする」── これらは機能の状態と振る舞いを定義しており、すでにUXの領域です。 UX要件の多くは、ハッピーパスの機能のあいだ（空・読み込み中・エラー・フォーカス順・画面幅での崩れ・言葉づかい）に宿ります。そしてそれらは、実装して画面に出して初めて見える ── だからこの教材の「触って体感する」やり方が、UXにこそ効きます。

    UX要件そのものは、上流の M3（誰の・どんな体験）で本格的に扱いました。機能要件の受け入れ条件（0件・状態）にも、UXは自然と滲んでいます ── 上流と要件は、いつも隣り合っています。

    次へ

    体験ループを1周まわしました。曖昧な依頼 → 握って委ねる壁打ち → 検証可能な成果物 → 実装 → 検収と振り返り。これがすべてのモジュールの背骨です。

    次の M5（情報設計＆コンテンツモデル）では、今回「種」として触れたコンテンツモデリングを、サイト全体の情報設計として深く掘ります。型の質が機能の質を決めることを、もう一段大きなスケールで体感します。

    📨 次の依頼が、もう届いている

    デジタル・三宅から R-02：「ジャーナルを成分・効能で探せるようにしたい」。一覧に“絞り込み”を足すだけに見えて、実は記事の型そのものを問う依頼です ── プレスリリースで category を入れ忘れたあの感覚を、思い出すことになります。M5 で向き合います。
