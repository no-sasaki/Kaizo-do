改造道 ／ Phase 3 ・ To-Be 体験定義

M3 ── 誰の・どんな体験（T4／T7／T8）｜ 体験原則は深く

「親切さ」は、
状態と振る舞いに宿る。

体験原則（design principles）は、壁に貼ったポスターのままだと何も変えません。 原則が本物になるのは、読み込み中・空・エラーといった“状態”の振る舞いに翻訳されたとき。 言語化しにくいUXを、検証できる要件に落とす回です。

この回を終えると、「親切さ」のような曖昧な体験を、状態ごとの受け入れ条件として書けるようになります。

① 依頼を受ける

M1で目的、M2で現状の課題が見えてきました。次は、目指す体験を言語化します。そんな折、ブランドを預かる CMO・佐倉さんから、つかみどころのない相談が来ます。

💬 CMO・佐倉 怜 からの依頼

「見た目はすごく綺麗になったの。でもね、実際に使うと“親切さ”が足りない気がするのよ。 うちのブランドって『静けさ』でしょう？ その世界観が、細かい体験のところで裏切られてる瞬間がある。ちゃんと“体験”を設計したいの。」

機能要件（M4）のように「何ができるか」を言えばいい依頼ではありません。これは 体験（UX）の依頼。曖昧で、言語化しにくく、放っておくと「いい感じにします」で終わってしまう。M3 のゴールは、この曖昧さを検証できる体験要件に変えることです。

② AIと壁打ちして要件を定義する

ペルソナとジャーニーは“最小”で素早く（AIに広く委ねる）。そのうえで本当に握るのは、体験原則を「振る舞い」に翻訳するところです。ここはAIに任せきりにできません ── 実装でどんな状態が起きるかが見えていないと、原則がポスターのまま終わるからです。

▷ AIに委ねる

  • 優先ペルソナ・ジャーニーの草案
  • タッチポイントと感情の起伏の洗い出し
  • 体験原則（design principles）のたたき台
  • 各画面で起こりうる状態の列挙（loading/empty/error…）

● 人が握る

  • 効きどころ（重点モーメント）の特定
  • ペルソナが意思決定を変える粒度か
  • 体験原則を検証可能な振る舞いへ翻訳
  • ブランド（静けさ）との整合

握ったら、書く ── ここで決めた判断は、状態ごとの振る舞い要件（空・読み込み中・エラー…）に書き留めて初めて「握った」ことになります。

まず“発散”。ペルソナ・ジャーニー・原則・状態を一気に出させます。

🧭 コンサルの思考発散：誰の・どんな体験・どの瞬間
あなたは経験豊富なUX／体験設計のコンサルタントです。
京都発のウェルネスブランド「NAGI」のブランドサイトの体験を設計します（非EC、多言語）。
CMOの相談は「見た目は綺麗だが“親切さ”が足りない。静けさの世界観が細部で裏切られる」。

私が判断するための“発散”をしてください。
1. 優先すべきペルソナを1〜2人（意思決定を変える粒度で。属性の羅列でなく、目的・不安・文脈）。
2. そのペルソナの主要ジャーニーと、感情が動く「重点モーメント（効きどころ）」を1つ。
3. ブランド「静けさ」を体験に落とす体験原則（design principles）を3〜5個。
4. サイトの主要なUIで起こりうる“状態”を列挙（入力前/読み込み中/成功/空(0件)/エラー/不正入力/低速回線 等）。
5. 「綺麗なのに不親切」になりがちな具体例を、状態の観点で3つ。
最後に、私が必ず自分で決めるべき論点を3つ。

ここで握るのは、原則を「言葉」で終わらせないこと。「静けさ」を“待たせない・迷わせない・急かさない・隠さない”という振る舞いにまで降ろします。次の収束プロンプトで、検証可能な要件に翻訳させます。

🧭 コンサルの思考収束：体験原則 → 振る舞い要件
方針を確定します。
- 重点モーメント：見込み客がジャーナルを読んで信頼し、ニュースレター登録で関係を始める瞬間。
- 体験原則：「待たせない」「迷わせない」「急かさない」「隠さない（正直）」。
- 成功の定義：どの状態(入力前/送信中/成功/エラー/不正入力)でも、ユーザーが次の一手に迷わない。

各体験原則を、検証可能な「状態ごとの振る舞い要件（受け入れ条件 Given/When/Then）」に翻訳してください。
特に loading / success / empty / error / invalid の各状態について、
「何を表示し」「何を案内し（次の一手）」「どんな言葉（マイクロコピー）で」を具体化すること。
曖昧語（"親切に"等）は禁止し、観察・テストできる記述にすること。MoSCoWで優先度も。

● 出力は一例 返ってきた翻訳を、ブランドの「静けさ」と成功の定義に照らして取捨するのが握りです。プロンプトを自分で書けるようになるには →（Appendix）

                                     （送信中・成功・エラー・不正入力の振る舞い）

力を注ぐのは全部ではなく、この1点。慎重派の美和が「このブランドは信頼できる」と判断する瞬間。

成果物C ｜ 体験原則 → 振る舞い要件（M3の核心）T8
体験原則状態検証可能な振る舞い要件
待たせない読み込み中送信後、即座にボタンを無効化＋「送信中…」を表示（無反応の時間を作らない）。
低速回線スケルトン/プレースホルダで知覚速度を保つ。
迷わせない空(0件)「該当なし」だけで終わらせず、必ず次の一手（解除・別提案）を出す。
隠さない（正直）エラー失敗を隠さず、理由と再試行導線を出す。楽観的UIで成功を偽装しない。
急かさない不正入力送信を黙って通さない。インラインで責めない言葉のエラー＋該当項目にフォーカス。
静けさ（全体）成功派手に祝わず、静かな確認＋次の体験（記事へ）をそっと差し出す。

この表が成果物の心臓。原則（左）が、テストできる振る舞い（右）になっている＝設計者が手を止めない。

UX要件は、ハッピーパスの“あいだ”に宿る

機能要件は「正常に動いたとき」を語ります。けれど実際のユーザーは、そのあいだに落ちます ── 読み込み中、結果ゼロ、通信エラー、入力ミス。ここをどう扱うかが「親切さ」の正体で、実装して画面に出して初めて見える。だからUXは、座学より“触って体感”が効きます。

「綺麗なのに不親切」の多くは、デザインの失敗ではなく “状態”の設計漏れ。ハッピーパスだけ作り込み、loading/empty/error/invalid を放置した結果です。

少しズームアウトすると、UXは大きく2つの軸で捉えられます。このモジュールは 軸1（振る舞い）を深く扱い、軸2（知覚・構造）は、それが最も“見える・測れる”場所であとのモジュールで実践します。

軸1 ── 振る舞い（時間の中の体験）

状態・フィードバック・流れ

読み込み中／空／エラー／入力ミスをどう扱うか。粒度にも依らず効く軸で、このモジュール（M3）で深く扱います。

軸2 ── 知覚・構造（一目で受け取る体験）

階層・可読性・アフォーダンス・密度・操作のしやすさ

「見やすい・操作しやすい」の正体。M2 で実装を開いて“診断”し、M6 で“測る”（コントラスト・フォーカス・タップ領域・知覚速度）。

どちらの軸も共通するのは ──「良い体験」という曖昧語を、検証可能な要件に翻訳すること。軸が変わっても、その翻訳の作法は同じです。

なぜ「振る舞いへの翻訳」にこだわるのか ── 原則だけでは、体験は変わらない

③で書いた「振る舞い要件」が、体験をどれだけ左右するか。実装に入る前に、同じ画面・同じ原則で 状態を要件化した場合としない場合を見比べます。

● 以下は一例 見るべきは「不親切がどこで生まれるか」です（このあと自分でも触れます）。

✗ 体験原則を“言葉”で止めると

要件：「静けさのある親切なフォームにする」（=ハッピーパスだけ実装）

  • 🔴 送信中：無反応（押せたのか分からず連打）
  • 🔴 エラー：何も起きない（成功と区別がつかない）
  • 🔴 不正入力：黙って通る（あとで届かない）
  • 🔴 空(0件)：行き止まり、次の一手がない

“綺麗なのに不親切”が完成する。原則は壁のポスターのまま。

✓ 原則を“振る舞い”に翻訳すると

要件：状態ごとの受け入れ条件（成果物C）

  • 🟢 送信中：即フィードバック＋ボタン無効化
  • 🟢 エラー：理由＋再試行（隠さない）
  • 🟢 不正入力：やさしいインラインエラー＋フォーカス
  • 🟢 空/成功：必ず次の一手をそっと提示

静けさが、細部まで届く。差を生んだのは美しさではなく、状態の要件化。

自分で触って確かめる ── 状態シミュレータ

ニュースレター登録フォームを題材に、「原則だけ（状態を作り込まない）」と「振る舞いまで要件化」を切り替えながら、各状態を実際に表示してみてください。同じUIでも、状態の扱いで“親切さ”がここまで変わります。

    ⚙️ 開発体験用・コピペでOK実装：状態まで作り込んだニュースレター登録
    あなたはフロントエンド実装者です。静的サイト「NAGI」に、ニュースレター登録フォームを追加します。
送信先は無いので、送信はクライアントで擬似的に処理（成功/失敗をシミュレート）して構いません。
デザイントークンは assets/css/nagi.css を踏襲し、ブランドの「静けさ」に合う控えめな表現にすること。

次の“状態ごとの振る舞い要件”をすべて満たすこと（ハッピーパスだけにしない）：
- 入力前：ラベルと、月1回配信である旨の控えめなマイクロコピー。
- 送信中：送信ボタンを無効化し「送信中…」を表示（aria-busy）。無反応の時間を作らない。
- 成功：派手に祝わず、静かな確認文＋「最新のジャーナルを読む」への導線。
- エラー：失敗を隠さず、理由と「再試行」導線を表示。
- 不正入力（メール形式不正）：送信を通さず、該当項目にインラインの“責めない”エラー文＋フォーカス移動。aria-invalid を付与。
- アクセシビリティ：ラベルと入力の関連付け、エラーは aria-live で通知。

    気づき 1

    UXは“あいだ”に住む

    機能が正常に動く瞬間ではなく、loading/empty/error/invalid の扱いに体験の質が出る。ここは実装を見ないと書けない。

    気づき 2

    原則は“振る舞い”で測る

    「静けさ」を、待たせない・隠さない・急かさないという観察可能な振る舞いに降ろせば、合意も検収もできる。

    十分性チェック

    セルフチェック合意の前に
    • □ ペルソナは意思決定を変える粒度か（属性の羅列で終わっていないか）
    • □ 力を注ぐ重点モーメントを1つに絞れたか
    • □ 体験原則が検証可能な振る舞いに翻訳されているか（"親切に"が残っていないか）
    • □ loading / success / empty / error / invalid の各状態に要件があるか
    • □ 各振る舞いがブランド（静けさ）と整合しているか
