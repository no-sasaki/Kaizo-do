なぜこの教材か

経営課題を整理し、To-Be を描き、ロードマップに落とす ── 戦略の設計は、コンサルティングの確かな価値です。そこに“実装が見える目”を一つ足すと、要件の解像度が変わります。 上流で決めたことが、下流の実装でどう効くかが見えるようになるからです。

決めきれていない要件は、最も高くつく場所＝実装フェーズで初めて表面化します。「カテゴリで絞り込めると思っていた」「多言語は後から足せると思っていた」── こうしたズレは、設計図の上では美しく、コードの上で崩れる。現場で誰の身にも起きる構造です。

この教材の核心は、突き詰めるとたった一つ。
要件定義の質は、そのままプロダクトの質になる。そして生成AIは、技術が“見えている”人が使うほど、要件定義を速く・広く・深くする増幅器になる。

必要なのは、ブラウザとチャットAIだけ

用意する道具は、たった二つです。

  • ブラウザ ── GitHub の Web エディタとプレビュー
  • チャット型の生成AI ── ChatGPT / Claude などの対話型AI

ローカル環境も、コマンドラインも、サーバも要りません。それでも、本物のブランドサイトを動かし、コンテンツの“型”を定義し、機能を実装し、結果を目で見られます。生成AIが、そこまでの学習コストを劇的に下げています。

作業環境の準備 ── ブラウザだけで完結

学習の「砂場」になるのは、仮想ブランド NAGI のサイトです。まずは触ってみましょう。

STEP 1

NAGI サイトを見る

下のボタンから、完成済みのブランドサイトを開きます。製品コレクション、ジャーナル記事 ── これらはすべて、裏側の /data/*.json という構造化データから描画されています。

NAGI サイトを別タブで開く ↗

STEP 2

自分のコピーを持つ（Fork）

本番の演習では、このリポジトリを GitHub 上で Fork（または “Use this template”）して、自分のアカウントにコピーします。設定 → Pages を有効にすれば、自分専用のNAGIサイトが数分で公開されます。編集はすべて GitHub の鉛筆アイコン（Webエディタ）から。git のインストールもコマンドも不要です。

※ まず読み進めるだけでも全体像をつかめます。手を動かす準備ができたら Fork してください。

STEP 3 ── 最初の一手（30秒）

コンテンツを1行変えて、サイトに映す

いきなりコードは触りません。Fork したリポジトリで nagi/data/products.json を Web エディタで開き、先頭の製品の "tagline_ja" の文字を好きな一文に書き換えて “Commit” するだけ。数十秒後、コレクションページのその製品の説明が、書いた言葉に変わります。

触ったのは コンテンツ（中身）だけ。表示のコード（見た目）には一切触れていないのに、画面が変わった ── これが「中身」と「見た目」が分離した構造の、最初の手応えです。本物のサイトを、自分の手で動かせました。

🧭 セットアップは、この一度だけ

Fork して自分の Pages で公開する ── 手を動かすために必要な準備はこれだけです。あとはブラウザの中だけで「編集 → 確認 → AIに相談」のループが回り続けます。

反映されない・間違えたとき

GitHub Pages への反映には数十秒〜数分かかることがあります。変えたはずなのに画面が変わらないときは、少し待ってから再読み込みしてください。 文字を間違えても、同じファイルをもう一度編集して Commit すれば上書きできます。大きく崩れてしまっても、GitHub の Commit 履歴から前の状態に戻せますので、安心して試せます。

学びの舞台 ──「凪 / NAGI」

NAGI（凪）は、京都発・グローバル展開中の架空のプレミアム・スキンケア＆ウェルネスブランドです。製品コレクション、成分・科学のストーリー、リチュアル、サステナビリティ、ジャーナル（編集記事）── 多様なコンテンツを持つ、非ECのブランドサイトです。

NAGI ブランドブリーフ（抜粋）与件

パーパス：過剰を削ぎ落とし、肌が本来もつ静けさを取り戻す。

世界観：「凪」＝波が引いたあとの鏡のような水面。静謐・余白・上質。

素材：京都・伏見の発酵技術と瀬戸内の柑橘。土地の素材 × 科学。

コンテンツ領域：製品コレクション／成分・科学のストーリー／リチュアル（How-to）／サステナビリティ／ジャーナル（編集記事）／（今後）店舗・キャンペーン・プレスルーム。

市場：日本（ja-JP）と海外（en-US）。地域差・表現規制の差がガバナンスの論点になる。

このブリーフを読んで気づいてほしいのは、コンテンツの“種類”が豊富であること。製品、記事、リチュアル、プレス…… これらはそれぞれ違う「型」を持ちます。WCMの心臓＝コンテンツモデリングが自然に必要になる舞台です。

プロジェクト憲章（T1）── 何のための・誰が決める・どこまで

Phase 0 の役割は、プロジェクトの輪郭を最初に固定することです。ここでは与件として提示します（深く作るのは後のモジュール）。

NAGI サイト・リニューアル ｜ プロジェクト憲章（要約）T1 OUT
目的ブランド世界観を保ちながら、語れる・見つかる・運用できるブランドサイトへ。海外市場での発見性とガバナンスを強化する。
背景現行サイトは静的でコンテンツ追加が属人的。海外PR・多言語の要望が増加。
スコープ仮説WCM（コンテンツ管理・表現・情報設計）に集中。EC機能は対象外。
体制（RACI抜粋）意思決定：CMO（A）／要件定義：担当コンサル（R）＝この案件で動く役回り／承認：CMO・法務（C/A）／運用：デジタル（R）
制約技術前提として静的配信（GitHub Pages 相当）＋構造化データ駆動。多言語・ガバナンス必須。
進め方要件定義の15タスクを5フェーズで。各機能は「依頼→要件→実装→結果」の体験ループで検証する。

要望バックログ ── 難度は徐々に上がる

ステークホルダーからの要望はバックログ化され、モジュールが進むごとに難度が上がります。後半の M4 からは、これらの具体的な依頼を「依頼 → 要件 → 実装 → 結果」の体験ループで扱います。最初に向き合う代表例が、PR・林さんからの依頼（R-01）です。

常設フレーム ──「AIに委ねる／人が握る」

これから全モジュールで、この線引きを反復します。最初に頭に入れておいてください。

▷ AIに委ねる

  • 選択肢・草案・たたき台の量産
  • エッジケース・抜け漏れの洗い出し
  • 定型の構造化（表・リスト・JSON）
  • 実装コードの生成

● 人が握る

  • 成功の定義（何をもって良しとするか）
  • 体験原則・ブランド制約との整合
  • 合意可能性（関係者が揉めないか）
  • 優先度のトレードオフと十分性の判断

AIは速い。だが「速く間違える」こともできます。だからこそ、判断＝握りの部分が人の価値になる。これがAI時代のコンサルタントの筋力です。
