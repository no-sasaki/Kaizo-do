# あなたへの依頼


日本のコンサルティング業界が超上流、上流工程に加えて、技術や実装の知識つけて深みのあるコンサルテーションが行える無料の教育サイトを作って。GitHub Pagesで公開。デザインもUI/UXも最高を頼む。プログラミングやコマンドラインに触ったことがなく青写真を描くようなコンサルテーションしかしたことがないようなコンサルタントに技術的知見や実装経験を理解すること、経験することで深いコンサルテーションが提供する出来ることの腹落ちをし、しかしながら今の時代は生成AIを最大限活かすことで、学習コストを極限まで抑えつつ価値提供を何倍にも出来る、加えて、技術的知見や実装経験を踏まえてより深く広い視点での要件定義が生成AIを活用することで高速かつ高品質で行えることが伝わることようにすること。


コンサルテーション領域はデジタルマーケティングのweb content management領域に絞ります

学習する上で仮想のエンタープライズ企業・ブランドサイトを設定し、Github pagesで動作する仮想のブランドサイトも用意しておく。
学習者はこのブランドサイトに対する追加要件を要件定義・実装することでケーススタディとして実践的な学習が行えるように設計する。

また単に開発タスクの表面に触れるだけでなく、実案件での要件定義がプロダクトの品質にどのように直結するのかを身を持って体験できるように、実際に要件定義し成果物も作成する。
ただし、成果物のファイル形式にはこだわらず、本質的に重要な情報や観点が網羅されていることを重要視する

## 1. 全体コンセプト ── 何を腹落ちさせるか

設計の背骨は3つの転換です。

**「技術を学ぶ」を目的にしない。** ゴールはコードが書けることではなく、**技術が見えることで要件定義の解像度が上がること**。技術経験はあくまで「より深い要件定義」のための手段だと一貫させます。これを外すと、ただの初心者向けプログラミング教材に堕ちます。

**制約（GitHub Pages＋Chat AIのみ）を"弱点"でなく"証明"として使う。** ローカル環境もコマンドラインも使えない人が、ブラウザとチャットAIだけで本物のブランドサイトを動かし、コンテンツモデルを定義し、機能を実装し、結果を目で見る。**これができてしまうこと自体が、「AIが学習コストを極限まで下げる」という主張の生きた証拠**になります。制約は教材の敵ではなく、最大の説得材料です。

**AIへの「委ねる／握る」を全タスクで明示する。** いただいた表の思想 ── AIに量産・草案・実装を委ね、人は「成功の定義」「合意可能性」「十分性」を握る ── を、各モジュールの常設フレームとしてUIに埋め込みます。受講者は毎回「ここはAIに投げる／ここは自分が責任を持つ」の線引きを反復し、それが習慣になります。

## 2. 体験の中核 ── 5ステップの「体験ループ」

各ハンズオンは、次のループを1周します。座学→演習ではなく、**毎回このループを回すこと自体が学習**です。

**①依頼を受ける** … ステークホルダーから、現実的に少し曖昧な要望が来る（例：「海外の記者にプレスリリースをちゃんと見せたい」）。
**②AIと壁打ちして要件を定義する** … チャットAIにユーザーストーリー・受け入れ条件・コンテンツモデルの草案を量産させる。ただし判断（成功の定義／合意可能性／優先度）は受講者が握る。
**③成果物をつくる** … ストーリー＋受け入れ条件、コンテンツモデル、トレーサビリティ表などを実際に書き起こす。
**④AIに実装させる** … 成果物をAIに渡してコードを生成させ、自分のブランドサイトに反映する。
**⑤結果を体験し、振り返る** … 動いたサイトを見る。要件が鋭ければ実装は滑らかで品質が高い。観点が抜けていれば、壁にぶつかる・手戻りする・エッジケースで壊れる。そして「②の質が⑤の品質を決めた」を明示的に言語化する。

このループの肝は、**意図的に「浅いやり方」と「深いやり方」を両方踏ませる**こと（後述の「わざと失敗させる」設計）。読むだけでは絶対に得られない、品質差の体感がここで生まれます。

## 3. 学びの舞台 ── 仮想ブランド「凪 / NAGI」

WCMの概念（多言語・多地域、ガバナンス、複数コンテンツタイプ、ブランド表現）が**全部"自然に必要になる"舞台**を一つ用意します。提案は ──

**NAGI（凪）** … 京都発、グローバル展開中の架空のプレミアム・スキンケア＆ウェルネス・ブランド。
- ブランドの世界観・ヘリテージ（ブランド要求T3が効く）
- 製品コレクション、成分・科学のストーリー、リチュアル（How-to）、サステナビリティ、ジャーナル（編集記事）、店舗・スパ検索、キャンペーン（＝豊富なコンテンツタイプ＝T10の心臓）
- 日本／海外の複数市場（i18n・地域差・表現規制の差＝ガバナンスとT11/T12が効く）

ブランドサイト（非EC）に絞ることで、トランザクションの複雑さを避けつつ、**ブランド表現とコンテンツ運用というWCMの本丸**に集中できます。

舞台には**架空のステークホルダー**も配置します。CMO（曖昧なブランド起点の要望）、デジタルマーケ責任者（KPI起点）、地域マネージャー（ローカライズ要求）、法務・コンプラ（ガバナンス）、IT/アーキテクト（技術制約）。彼らが「追加要件」を投げてくる存在になり、ケーススタディに血を通わせます。要望はバックログ化しておき、モジュールが進むごとに難度が上がる構成にします。

> ドメインは差し替え可能です。B2B SaaS／食品・飲料／アパレル／家電なども検討余地あり。特にご指定なければ NAGI で進めます。

## 4. 技術アーキテクチャの肝 ── JSON駆動の「擬似ヘッドレス」

ここが計画で一番重要な発明です。**「GitHub Pages＋AIだけで、どうやってコンテンツモデリング（T10）を体験させるか」** ── ご自身が「重要だが学習環境の用意が難しい」と書かれた一点への答えです。

NAGIサイトを、**構造化データ（JSON）＋クライアントサイドのレンダリング**で作ります。

- `/data/*.json` … 構造化コンテンツ（＝「CMSの中身」に相当）
- 素のJS … JSONを読み込んでHTMLに描画（＝「表示層」に相当）
- ビルド工程ゼロ。GitHubのWebエディタで編集でき、GitHub Pagesでそのまま見られる

これは小さな「ヘッドレスCMSのミニチュア」です。受講者は、**コンテンツモデル（型）と表現（見た目）が分離している状態を、自分の手で体験**できます。新コンテンツタイプ（例：カテゴリ付きプレスリリース）を追加する課題では、

1. 型を定義する（プレスリリースに必要なフィールドは？ 公開日は？ カテゴリは？ 多言語は？）
2. JSONでインスタンスを作る
3. 描画・絞り込みを実装する
4. **型を雑に決めていると実装で壁にぶつかる**（カテゴリ項目を型に入れ忘れた→絞り込みが作れない、等）

を一気通貫で踏みます。**コンテンツモデルの質が、機能の質に直結する**ことを、抽象論ではなく自分のサイトの不具合として食らう。これがT10の攻略法です。さらに、自分で「擬似ヘッドレス」を建てた経験があるからこそ、M7（ヘッドレス vs 従来型 vs DXPの方式判断）が体感を伴って理解できる、という波及効果も狙えます。

そして**受講者の作業は完全にブラウザ完結**です：GitHubのWebエディタで編集 → GitHub Pagesで確認 → チャットAIでコード・コンテンツ生成。git CLIもローカルサーバも一切不要。この「ブラウザだけで回る」ことを、私が実際に検証した上で組みます。

## 6. 実験設計 ── 要件の質を“見える化”する仕掛け

主張を本当に腹落ちさせるには、**要件の質が結果をどれだけ動かすかを、観察できる実験にする**のが最短です。◎モジュールにはこの実験を仕込みます ──

粗いブリーフ（曖昧な受け入れ条件・雑な型）と、精緻なブリーフ（検証可能な受け入れ条件・効いている型）を**同じAIに渡して出力を見比べる**。粗い方は「それっぽいが壊れている」もの（エッジケースで崩れる、多言語で破綻する、運用で詰む）に、精緻な方は一発で通る。**同じAI・同じ依頼でも、ブリーフの解像度で結果がここまで変わる**を一度目の当たりにすれば、主張は説教ではなく自分の記憶になります。これが座学との決定的な差です。

> **重要（学習者向けの言い回し）**：この設計の制作意図は「浅いやり方の失敗を見せる」ことだが、**学習者に向けては「あなたを試す／失敗させる」と書かない**。「同じAIに2つのブリーフを渡して観察する実験」として、目的を先に示し、主役は学習者・わかるのは“要件のレバレッジ”という empowering なフレームにする（→ 横展開の制作規約 C）。

## 7. デザイン / UX 方針

「最高」のご要望に対し、**二つのデザインシステムの対比**で応えます。

**学習プラットフォーム側** … エディトリアルで知的、余白広め、強い情報階層。和文＋欧文の上質な組み合わせ、インク／ペーパー基調＋洗練されたアクセント一色。「依頼→要件→実装→結果」を見せる分割レイアウト、コピー可能なプロンプトカード、成果物テンプレート、進捗と「今ここ」の道標。落ち着いた"コンサルティングファームの読み物"のトーン。

**NAGIブランドサイト側** … プレミアムでブランドらしい、まったく別物の世界観。受講者が「本物の美しいブランドを触っている」と感じることで、品質への当事者意識が上がります。

この対比自体が教材です ── **ブランド表現のUIと、機能的なUIは別物**だと体で分かる。両方とも完全静的・高速・アクセシブルに作り、「言行一致（自分たちが非機能を守っている）」を示します。実装時には frontend-design スキルに従って作り込みます。

## 8. リポジトリ / 公開構成

- **教材サイト**：私たちが公開する読み物（受講者は読むだけ）。
- **NAGIスターターサイト**：受講者が **Fork（または "Use this template"）** して自分のGitHub Pagesに持ち、Webエディタで改造する"砂場"。

この「Forkして自分のPagesで改造する」フローだけが唯一のセットアップ摩擦なので、Course 0でブラウザ完結のまま手取り足取り通します。私の側で**実際にこの導線が機能するか検証**してから本実装に入ります。

## 9. 制作ロードマップ

全部を一度に作るより、**まず一本"縦に"通して全体体験を確かめる**ことを強く推します。

- **Phase A（土台と証明）**：教材サイトのIA／NAGIワールド設定（ブランド・ステークホルダー・要望バックログ）／**NAGIスターターサイト本体（JSON駆動・美麗）**／Course 0

**高橋 亮介 / 32歳 / シニアコンサルタント**

## 所属・役割

大手コンサルティングファームのデジタル戦略・CX領域所属。

> 事業として何を目指すかは整理できます。

## 内心の不安

* 実装者に浅いと思われたくない
* クライアントのIT部門との会話で置いていかれたくない
* 自分の要件定義が後工程で手戻りを生んでいないか不安
* 生成AIを使えば何とかなる気もするが、どう使えば実務品質になるのか分からない

## この教材での学習ゴール

* WCM案件で設計者・実装者が必要とする要件の粒度を理解する
* 要件定義とコンテンツモデル、CMS運用、実装品質のつながりを体験する
* AIを使って要件を深掘りし、レビューし、実装理解を補助できるようになる

## 教材設計上の配慮

この人には、最初からコードを書かせすぎると離脱します。
まずは「あなたの得意な上流資料が、実装でどこまで通用するか」を体験させるのがよいです。

向いている導入：

```text
あなたが書いた『問い合わせ導線を強化する』という要件は、
実装者から見るとまだ曖昧です。
では、どこが曖昧なのかを一緒に分解しましょう。
```

# ボイス＆トーン原則（受講者向けコピーの絶対規律）

ペルソナ（高橋亮介）は、上流のプロとして既に有能で、内心「実装者に浅いと思われたくない」という不安を抱えている。
作る側の内部フレーム（例：「青写真しか描けない」）は **絶対に受講者へ直接伝えない**。
教材の役割は、欠点を指摘することではなく、潜在的な弱みを克服するきっかけを与えること。
そして「腹落ち」と同じくらい **「どんどん引き込まれていくか（読み物としての面白さ）」を重視する**。

## 6つのボイス原則

1. **まず肯定する** … 構想・To-Be・ロードマップ・KPI・論点整理は本物の武器。毎モジュール、それを出発点として扱う。
2. **「欠陥の修復」でなく「射程の拡張」** … `弱点 / 穴 / 苦手 / 盲点 / 塞ぐ` を受講者本人に向けて使わない。新スキルは「構想を実装まで生きたまま届ける増幅器」として提示。
3. **原因は本人でなく“翻訳”に外在化** … 詰まりは能力不足でなく「上流と下流が少し違う言語を話す」構造の問題。手戻りは誰にでも起きる。
4. **失敗は本人のものでなく「実験の発見」** … わざと失敗は“受講者を試す”のでなく“同じAIに2つのブリーフを渡すと何が起きるかを観察する実験”。主役は受講者、わかるのは「要件のレバレッジの大きさ」。empoweringに。
5. **Push でなく Pull（引き込む）** … 案件の主人公としての没入／早い成功体験／好奇心を煽る問い／モジュール間のクリフハンガー。読み物として面白いことを品質の一部とする。
6. **不安には必ず出口を同じ場所で** … 不安に触れるなら、その場で「これで超えられる」を示す。不安だけ置かない。

## 語り口の絶対ルール（高橋さんと二人きりの部屋）

ペルソナを設定して開発するとは、目の前に生身の高橋さんが居て、二人きりで話している状況。だから：

1. **部屋で二人** … メッセージ内で `あなた／あなたの` を使ってペルソナを名指し・描写しない。
   日本語の主語省略を活かし、**一般則**と**直接の手順**で語る。
   - 属性の描写 → 本人を指さず（×「あなたは戦略を描けるプロ」 ○「戦略の設計は、コンサルティングの確かな価値」）
   - 手順の指示 → 主語を落とす（×「あなたのSlackに届きました」 ○「Slackに、林さんからメッセージが届きました」）
   - 所有 → `あなたの砂場` ではなく `自分の砂場 / 自分のNAGIサイト`
2. **相手は我々を知らない** … ペルソナの属性に触れるときは本人を言い当てず「多くの場合」「現場では」と一般化する。
   知らない人にいきなり「あなたはこういう人ですよね」と当てられると、当たっていても不気味で不安になる。
3. **常に初対面** … 過去の表現を打ち消す・弁解する文（「〜を試すのではありません」等の前置き）を書かない。
   受講者は以前の版を知らないので、最初から肯定形で「何であるか」だけを書く。
4. **作り手のメタ戦略を見せない** … 制作側の意図・狙い・設計判断の“理由”を受講者向けに書かない。受講者が受け取るのは「なぜそう作ったか」ではなく「体験そのもの」だけ。次のような露出は禁止：
   - 設計都合の正当化 …「制約は“証明”」「WCMの“本丸”に集中するため」「ECの複雑さを避け」「学習コストを下げる主張の生きた証拠」「あえて〜に絞っている」等。狙いを語らず、結果だけを提示する（×「Aの複雑さを避けてBに集中するため非ECにしている」 ○「これは非ECのブランドサイトです」）。
   - 開発スラング …「縦切り」「縦切り版」「縦切りリリース」等は受講者にとって意味不明。「現在公開中」等に置換。
   - 作り手目線の独白 …「唯一のつまずきポイント」「腹落ちさせたい」のような、作り手が学習者を見ている視点の言い回し。
   - 自前の枠組みを一般知識のように見せない …「要件定義の0–5フェーズ」は**本教材独自の整理**（外部の確立した標準ではない）。「標準では〜」と匂わせず「**この教材では、要件定義をこの流れで学ぶ**」と、自前の枠組みだと伝わる言い回しにする。そして**フェーズとモジュールの対応を必ず明示する**（5フェーズと言いながら直後に10モジュールを並べると読み手が混乱する）。
   - **掲載前チェック**：すべての学習者向けテキストを「これは作り手から私への説明になっていないか？」とペルソナ視点で読み返す。

# 横展開の制作規約（縦切り M4 で確立した正典テンプレート）

縦切り（Course 0 → NAGI → M4）で確立した規約を、横展開（M1/M2/M3/M5/M7/M8/M9・M10）でも必ず踏襲する。
**参照実装は `course/m4.html`。迷ったら M4 に倣う。** 以下は「縦切りのフィードバックを横展開後も維持する」ためのルール集。

## A. モジュールページの構成（正典）
- `mod-hero`（crumb ／ eyebrow＝モジュール番号・対応タスク・深さ ／ h1 ／ lede ／ meta-row のタグ）
- 目次 `rail`（`layout-rail`。本文 `prose` は読み幅 **820px・中央寄せ**）
- 本文は体験ループの見出し構成に沿う
- 末尾：十分性チェック → 「次へ」＝**次モジュールの依頼を予告（クリフハンガー）**
- 共通ナビ：Course 0 / M4 / Appendix / NAGI サイト

## B. 体験ループ（◎の深い回ほど厳密に1周）
①依頼 → ②AIと壁打ち（委ねる/握る）→ ③成果物 → ④実装 → ⑤検収・振り返り。
- **①〜③＝要件定義とその成果物。④＝実装**（受講者がAIに委ねて“体験”する工程。コンサルがコードを書くという意味ではない）。**⑤＝受け入れ条件での検収**。
- 各モジュールに「▷ AIに委ねる ／ ● 人が握る」カードを必ず置く（委ねる/握るの反復）。

## C. デモ・実験は「目的先行」＋ empowering
- 浅い→失敗の対比は強力だが、**学習者に向けては「試す/失敗させる」でなく「観察する実験」**として提示する（主役は受講者、わかるのは要件のレバレッジ）。
  ※ 本ドキュメント section 6「わざと失敗させる」は**制作者の設計意図**。学習者向けの文言は必ず実験フレームに変換する。
- 実験・デモは必ず**「なぜ今これを見るのか」を先に書く**（M4：③→④の橋渡しとして提示した例に倣う）。

## D. プロンプトカード規約
- すべてのプロンプトに**種別ラベル**：🧭 コンサルの思考（自分で書けるべき）／ ⚙️ 開発体験用・コピペでOK。
- コピーボタンを付ける。
- 教材に載せる**AIの出力は必ず「一例（実行ごとに変わる）」と明示**し、学習者の結果と一致させようとしない（合わせ込み作業をしない）。
- 各モジュールから Appendix（`course/prompting.html`）へ「自分で書けるようになるには →」で誘導。

## E. NAGI 砂場の規約
- **コンテンツ（`data/*.json`）は摩擦を減らせる場合は最初から同梱**し、JSONへのリンクを示す。学習者は「中身づくり」でなく「要件→表示」に集中。
- NAGI は**本物のブランドとして美しく保つ**（没入感＝品質への当事者意識）。学びの装飾は教材側に置き、NAGI上は `learner-banner` のみ。
- **コードそのもの（構造化JSON・JSON-LD・alt/aria・data-analytics 等）を学習素材にする＝言行一致**。
- 「コンテンツの**型**」と「**表示**」の分離が中核の教育装置。型を変えると機能が変わることを毎回体感させる。

## F. インタラクティブ（Playground）規約
- playground 的な体験は**独立ページにせず、効くモジュールのページ内に埋め込む**（`.pg*` を `platform.css` で再利用）。
- 全モジュール必須ではない。要素のないモジュールがあってよい。

## G. UX の扱い
- UX要件は**「状態と振る舞い」**（空・読み込み中・エラー・フォーカス順・レスポンシブ・マイクロコピー）に宿る。**実装して初めて見える**ので「触って体感」で扱う。
- 本格導入は **M2**。各モジュールでも「機能要件のとなりにUXがある」ことを示す（M4末尾のブリッジに倣う）。
- **UXは2軸**（M2に「2軸地図」を掲載済み）：
  - **軸1 振る舞い**（状態・フィードバック・流れ）＝ **M2 で深く**。
  - **軸2 知覚・構造**（階層・可読性・アフォーダンス・密度・操作性）＝ **M3 で“診断（見る）”・M8 で“測る”**。リッチな PDP が実演キャンバス。
  - 共通の転移スキルは「曖昧な“良い体験”を検証可能な要件に翻訳する」。軸・粒度に依らず効く。

## H. デザイン・読みやすさ
- 2つのデザインシステム（教材＝エディトリアル／NAGI＝プレミアム）を**混ぜない**。
- 教材本文の読み幅は **約820px・rail込みで中央寄せ・1180px基調**。**homepage を品質の基準**とする。
- 自分たちが非機能（静的・高速・アクセシブル）を守る。

## I. 引き込み（Pull）
- 冒頭は**肯定から**（一般則で。名指ししない）。早い小さな成功体験を置けるなら置く（Course 0 の30秒の勝ちに倣う）。
- **ステークホルダーと要望バックログ（R-01…）で連続性**を作り、難度を逓増させる。
- モジュール末で**次の依頼を予告**（クリフハンガー）。
