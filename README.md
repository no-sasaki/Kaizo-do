# 改造道 ｜ Kaizō-dō

技術と実装を“体で”理解し、生成AIで要件定義を速く・広く・深くするための、無料・実践型の教材です。デジタルマーケティングの **Web Content Management（WCM）** に絞り、ブラウザとチャット型の生成AI だけで、一本道を完走できます。

構想を描く力は、それだけで確かな価値です。この教材は、その力を実装に届くところまで連れていく道具として、自由に使ってもらえたら嬉しいです。

## どう学ぶか ── 体験ループ

座学ではありません。各モジュールで、次の5ステップを一周します。

1. **依頼を受ける** ── 架空のステークホルダーから、現実的に少し曖昧な要望が届く
2. **AI と壁打ちして要件を定義する** ── ストーリー・受け入れ条件・コンテンツモデルの草案は AI に量産させ、判断（成功の定義／合意可能性／優先度）は自分が握る
3. **成果物をつくる** ── ストーリー＋受け入れ条件、コンテンツモデル、トレーサビリティ表などを実際に書き起こす
4. **AI に実装させる** ── 成果物を AI に渡し、生成されたコードを自分のサイトに反映する
5. **結果を体験し、振り返る** ── 動いたサイトを見て、要件の質が結果をどれだけ動かしたかを言葉にする

学習に必要なのは **ブラウザとチャット型の生成AI だけ**。ローカル環境もコマンドラインも要りません。GitHub の Web エディタで編集し、GitHub Pages で確認し、AI でコードを生成する ── すべてがブラウザの中で完結します。

各モジュールには、「**▷ AI に委ねる ／ ● 人が握る**」の線引きカード、コピーして使えるプロンプト、成果物のひな形を常設しています。

## 何を扱うか ── モジュール構成

舞台は、京都発の架空のプレミアム・スキンケア＆ウェルネスブランド **「凪 / NAGI」**（ブランドサイト。EC ではありません）。多言語・多地域、ガバナンス、複数のコンテンツタイプ ── WCM の論点が自然に必要になる世界です。学習順はモジュール番号のとおり（M0 → M8）。

| --- | --- | --- |

| `course/m1.html` | **M1** Why ＆ブランド。KPI の操作化、ブランド要求＝北極星 | 最小 |

ステークホルダー（CMO・デジタルマーケ責任者・PR・法務）が要望をバックログとして投げ、モジュールが進むごとに難度が上がっていきます。

## 二つの砂場 ── To-Be と As-Is

- `nagi/` ── リニューアル後の **NAGI（To-Be）**。改造していくブランドサイト。`nagi/data/*.json` がコンテンツの“型”と中身、表示層は素の JavaScript。
- `nagi-legacy/` ── **現行サイト（As-Is）**。M2 で診断する対象。ぱっと見はきれいでも、よく見ると不便が潜んでいます。

「コンテンツの**型**」と「**見た目**」が分離した状態を、自分の手で触って体感できる ── これがこの教材の中心の仕掛けです。型を雑に決めると、実装で壁にぶつかる。その手応えごと、設計図の上では見えなかったものが見えてきます。

## 技術メモ ── 「擬似ヘッドレス」

NAGI サイトは **構造化データ（JSON）＋クライアントサイド描画** で動く、ヘッドレス CMS のミニチュアです。ビルド工程はゼロ。GitHub の Web エディタで編集でき、GitHub Pages でそのまま表示されます。自分で小さなヘッドレスを建てた経験が、後半の方式判断（M7）を体感を伴って理解する土台になります。

教材側（エディトリアル）と NAGI 側（プレミアム）で、あえて異なるデザインシステムを用意しています。ブランド表現の UI と、機能的な UI は別物 ── その対比自体も教材の一部です。

## ローカルでの確認

静的サイトです。任意の静的サーバで確認できます（`fetch` を使うため `file://` 直開きではなく HTTP 配信が必要です）。

```sh
python3 -m http.server 8000
# → http://localhost:8000/
```

## 公開（GitHub Pages）

`.github/workflows/static.yml` がリポジトリ全体を Pages へデプロイします。`.nojekyll` により Jekyll 処理は無効化済みです。
