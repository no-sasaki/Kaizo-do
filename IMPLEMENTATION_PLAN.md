# 解像堂（Kaizō-dō）実装計画書 / IMPLEMENTATION PLAN

> 本書は **実装担当AIエージェント** 向けの自己完結な指示書です。
> エージェントが受け取る入力（引き継ぎパッケージ）：
> 1. **HANDOFF.md** … 入口。全体地図・読む順序・ビルド順・完了定義。**最初に読む。**
> 2. **本実装計画書（IMPLEMENTATION_PLAN.md）** … 「どう作るか」の正本。
> 3. **Curriculum Map（`curriculum-map.html`）** … 「何を作るか」の正本（モジュール／In・Out／深さ／モード）。
> 4. **BRAND_BRIEF.md** … 全モジュール共通の**唯一の走行シナリオ**（与件・ペルソナ・ジャーニー・ユーザーストーリー・受け入れ条件・コンテンツモデル・As-Is/To-Be）。設定はこれが正本。
> 5. **既存4ファイル**（`index.html` / `stage-1.html` / `define.html` / `curriculum-map.html`）… デザインと対話UIの**正本リファレンス**。必ず同梱する。
>
> 本書だけでも実装可能なよう、デザイントークン・コンポーネント・規約・受け入れ条件をすべて明文化してある。Curriculum Map に書かれたモジュール内容は**繰り返さない**。本書は「どう作るか」、Map は「何を作るか」、BRAND_BRIEF は「どんな設定で」を担う。

## 0. ミッション（最初に腹に入れること）

非技術者のコンサルタントが、**chatタイプの生成AIだけ**を使い、最小コストで実装の腹落ちを得て、エンタープライズ・ブランドサイトの「設計に渡せる要件定義」を、かつてない品質×速度で出せるようになる ── そのための教育商材を作る。

作るものは**2つのWebサイト**。どちらも **GitHub Pages 上で動く静的サイト**。

1. **解像堂（教育サイト）** … レッスン群。Curriculum Map の Course 0 / M1〜M10 に対応するページ。
2. **AOZORA珈琲 ブランドサイト（教材内で改修する実物）** … 事前に v1 を作っておく。「機能追加・改修が必要になった」という設定で、生徒が**このサイトに対して**要件定義→実装を行う。

中心思想は **「書いて合意」から「見せて合意」へ**。要件をAIで即・試作し、動くもので握る。教育サイトの体験も、最終的にこの一点へ収束させる。

## 1. 最重要：生徒の学習ループ（この体験のために全実装を最適化する）

生徒は IDE もコマンドラインも使えない。**chat型の生成AIのみ**。だから実装は次のループが**摩擦ゼロで回る**ことを最優先に設計する。

```
① 解像堂のレッスンを読む／ラボで要件を固める
② ブランドサイトの 1枚HTML を「丸ごと」コピー
③ chat AI に貼り付け、要件（受け入れ条件）とともに「実装して、1枚のHTMLで返して」と指示
④ AI が返した 1枚HTML を GitHub Pages のリポジトリにアップロード（ドラッグ＆ドロップ）
⑤ 公開URLで結果を確認 →「見せて合意」
```

この④が成立するには、**ブランドサイトが完全に自己完結した1枚のHTMLである**ことが絶対条件。②③が成立するには、**chatのコンテキストに収まるサイズ**かつ**AIが安全に改修できる構造**であること。これらは後述の制約に落とす。

## 2. アーキテクチャと技術制約（ハードルール）

これらは交渉不可。すべてのページが従う。

1. **静的サイトのみ。** ビルドステップ無し、バックエンド無し、バンドラ無し、フレームワーク無し（**Vanilla JS のみ**）。
2. **1ページ＝1つの自己完結 `.html`。** CSS は `<style>`、JS は `<script>` として**同一ファイルにインライン**。外部 .css / .js ファイルへの分割は**禁止**。
   - 唯一許可する外部読み込みは **Google Fonts**（`<link>`）。
   - 理由：①生徒がブランドサイトHTMLを丸ごとAIに渡して1枚で受け取れる、②GitHub Pages にファイルを置くだけで動く、③依存破綻が起きない。
3. **外部画像ファイル禁止（特にブランドサイト）。** 画像は CSS グラデーション／インライン SVG／data-URI で表現。これにより真に「1枚で完結」する。
4. **localStorage / sessionStorage に依存しない。** 状態は JS のメモリ内で持つ。（GitHub Pages 実環境では localStorage は動くが、ポータビリティとAI改修の安全性のため core 機能は非依存にする。使うなら任意・劣化許容で。）
5. **品質フロア（全ページ必須）**：レスポンシブ（モバイル〜デスクトップ）、キーボードフォーカス可視（`:focus-visible`）、`prefers-reduced-motion` 尊重、セマンティックHTML、十分なコントラスト。
6. **chat-AIフレンドリーな実装**：意味のある HTML 構造、セクションを示す日本語コメント、改修したい箇所が一目で分かる構成。ブランドサイトは特に、AIが壊さず改修できるよう「編集してよい領域」をコメントで明示。
7. **フォント**（Google Fonts、全ページ共通の読み込み）：
   `Shippori Mincho`（和文ディスプレイ）, `Fraunces`（欧文ディスプレイ）, `Noto Sans JP`（和文本文/UI）, `JetBrains Mono`（ラベル/コード/データ）。
   ※ ブランドサイトは別のフォント方針可（後述 §5）。
8. **言語は日本語**（UI・コピー・コメント）。
9. **対応ブラウザ**：最新の Chrome / Edge / Safari / Firefox。ベンダープレフィックス過剰実装は不要。

## 4. デザインシステム：解像堂（教育サイト）

教育サイトの全ページはこの一貫した視覚言語に従う。**正本は既存 `index.html` / `stage-1.html` / `define.html` / `curriculum-map.html`**。新規ページは下記トークンとコンポーネントをそのまま流用すること（各ページにインライン複製でよい）。

### 4.1 コンセプト
「青写真（ブループリント＝低解像度の青い図）を、解像度を上げてフルカラーの“実装”へ現像する」。写真の**ピント／焦点**を視覚言語にする。量産的な「クリーム地＋明朝＋テラコッタ」は**禁止**。**ブループリント・ブルーを主役色**に、写真の焦点（フォーカス・ブラケット）を構造装置に使う。

### 4.2 カラートークン（`:root` にそのまま定義）

**ライト（現像済み / developed）**
```
--paper:#EEEADD; --paper-2:#E6E1D1; --ink:#14202B; --ink-soft:#3C4A56; --ink-faint:#6B7681;
--blueprint:#103A5E; --blueprint-deep:#0C2A45; --blueprint-line:#9FB2BE;
--amber:#C9852F; --amber-soft:#E7B86A; --resolved:#2C6B59; --alarm:#B0492F;
--rule:#C7C0AC; --hair:#D4CEBC; --shadow:rgba(16,40,60,.14); --bg-grid:rgba(16,58,94,.07);
```
**ダーク（暗室 / darkroom）** … `[data-theme="darkroom"]` で上書き
```
--paper:#0C1722; --paper-2:#10202E; --ink:#E9E3D3; --ink-soft:#B7BFC6; --ink-faint:#7E8B94;
--blueprint:#5FA0D6; --blueprint-deep:#9CC6E8; --blueprint-line:#34536C;
--amber:#E6AE4E; --amber-soft:#F0C988; --resolved:#5FB59A; --alarm:#E07A5A;
--rule:#26394A; --hair:#1E3142; --shadow:rgba(0,0,0,.5); --bg-grid:rgba(120,180,225,.06);
```
- 背景は方眼グリッド（`--bg-grid` の linear-gradient、`background-size:34px 34px`）。
- アクセントの amber は**一閃**として少量。多用しない。

### 4.3 タイポグラフィ
```
--serif:"Shippori Mincho","Fraunces",serif;       /* 見出し・編集的リード・プルクオート */
--fraunces:"Fraunces","Shippori Mincho",serif;    /* ブランド表記・ドロップキャップ等 */
--sans:"Noto Sans JP","Helvetica Neue",system-ui,sans-serif; /* 本文・UI */
--mono:"JetBrains Mono",ui-monospace,monospace;   /* eyebrow・ラベル・データ・コード */
```
- `body{ font-feature-settings:"palt" 1; letter-spacing:.01em; line-height:1.8〜1.85 }`。
- 見出しは明朝で大きく自信を持って。本文は Noto Sans JP。ラベル類はモノ＋大文字＋トラッキング。**過剰なボールドは避ける**（編集メディアの品格）。

### 4.4 共通コンポーネント（既存ページから流用・命名そろえる）
- **`.site-head` / `.brand`**：sticky ヘッダ。ブランドは「解像堂 / Kaizō-dō」。右に**テーマトグル（暗室⇄現像）**。
- **`.eyebrow`**：モノの小ラベル＋先頭に短い罫。セクションの分類に。
- **`.bracket`**：フォーカス・ブラケット（左上・右下の角だけの枠）。カード/ラボの「焦点」を示す**署名的装置**。
- **`.lesson-hero` + `.lesson-lead`**：レッスン冒頭。パンくず（`解像堂 / STAGE xx`）＋明朝の大見出し＋明朝リード。
- **`.prose h2`（`.n` 付き番号）/ `.prose p`**：本文セクション。最大読み幅 `--measure:38rem`。
- **`.callout` / `.callout.aha`**：気づき枠（aha は amber 左罫）。`.k` ラベル＋明朝の腹落ち文。
- **`.ai-split`（`.ai` / `.self`）+ `.prompt`**：AI境界線（委ねる/握る）＋実物プロンプト枠（ブループリント地に等幅）。
- **`.return`（before/after）**：コンサル還元。Before（淡色）→ After（明朝・`AFTER`マーク）。
- **`.museum`**：失敗の博物館。斜線地＋amberラベル。
- **`.reveal`（フェイルセーフ付き）**：スクロール出現（blur＋translate→sharp）。**必ず下記フェイルセーフを入れる**（後述 §11）。
- **`.stage`（live/soon, tag now/soon）**：ステージ一覧の行。
- **`.nextcard` / `.back`**：レッスン末尾の次へ／戻る。
- **テーマトグル JS / reveal JS** は §11 の定型を全ページに複製。

### 4.5 モーション
- ヒーローの**ピントダイヤル**（ホーム）、スクロール出現（控えめ）、ホバーのブラケット。`prefers-reduced-motion` で全停止。過剰な演出はAI生成臭の元。**署名は1つ**（焦点/解像度）に集中。

## 5. デザインシステム：AOZORA珈琲 ブランドサイト（**別アイデンティティ**）

重要：ブランドサイトは**教育サイトと別の世界観**を持つ。解像堂のブループリント調を**流用しない**。これは「クライアントの実ブランド」であり、本物のスペシャルティ珈琲ブランドに見えること。

- **方向性**：上質・誠実・余白。温かみのあるスペシャルティ珈琲。ブランドステートメント「朝を、もう一段くっきりと」。
- **配色（提案。エージェントが微調整可）**：生成り/オフホワイト地、深いエスプレッソ・ブラウン（例 `#2A211B`）、温かいクリーム、アクセントに焙煎を想わせる赤茶〜琥珀。解像堂の `#103A5E` ブルーは使わない。
- **タイポ**：欧文セリフ（例 `Fraunces` か `Cormorant`）＋和文（`Shippori Mincho` か `Zen Old Mincho`）＋本文サンス。コーヒーブランドらしい上質さ。
- **画像は使わない**：豆・湯気・店舗の雰囲気は CSS グラデーション／インライン SVG／装飾で表現（§2 制約3）。
- 1枚完結・自己完結・chatに収まるサイズ（後述）。

## ◎ 学習フォーカス設計 ── 15タスクのうち、何を深く学ぶか（§6〜§7の前提・最重要）

本カリキュラムの学習ゴールの一つは「**最小労力で要件定義〜開発を一周し、全体像を俯瞰できる**」こと。全15成果物を均等に深掘りするのは設計ミスであり、**焦点を絞る**。本節は §7 のモジュール設計を統べる前提である。

### A. 設計フェーズに入る「必要十分」の定義（合意ゲートの中身）
設計者が解釈の余地で手を止めない状態 ＝ 次の5つが「曖昧でなく、検証可能な言葉で、関係者に合意されている」こと。M10 の合意ゲートはこれを判定する。
1. **なぜ作るか** … ビジネスゴール／KPI と ブランド要求
2. **誰のどんな体験を実現するか** … ペルソナ／ジャーニー／主要シナリオ
3. **何を満たすか** … 機能・非機能・コンテンツ・情報の要件（優先度つき）
4. **何の上に作るか** … プラットフォーム／CMS方針・外部連携・移行方針
5. **どこまでやるか** … スコープ・制約・前提・リスク・トレーサビリティ

### B. フォーカス設計の原則
- **深く学ぶ（FOCUS）**＝非Techコンサルの最大ギャップ × 人の判断が要る × 「設計に渡せるブランドサイト要件」の核 × 実装の腹落ち＋AIが効果を一変させる場所。
- **軽く通す（LIGHT）**＝コンサルの既存の上流強み、またはAIに大半を委ねられるタスク。
- **与件・参考（GIVEN/REF）**＝営業で把握済み（Ph0）、または本テーマ外（非機能）。
- 帰結：**焦点は Phase 4（要件への落とし込み）＋合意ゲートに集中**。Phase 1・3（なぜ／To-Be体験）は既存強みのため LIGHT（AIで加速）。Phase 0 は与件、非機能は参考。

### C. AI役割分担は「フォーカス設計」と一体（ここが学習の核心）
実プロジェクトでも、要件定義の各タスクで chat型生成AI を使い倒す。各モジュールは**そのタスク固有の「AIに委ねる／人が握る」を、実物プロンプトとともに必ず提示**する。さらに：
- **LIGHTタスク＝AI主導・人は検品**として実装（「委ねてよい」を体験）。
- **FOCUSタスク＝人主導・AI増幅**として実装（「人が握る」を体験）。
- **この対比そのものが、役割分担を自然に身につける教材になる。** 外すタスクは、実務でも「AIに大半を委ねてよい」タスクだと示すこと。

### D. 15タスク × 深さ × AI役割分担 × モジュール対応

### E. 「深く作る成果物」は4つに限定する
深く**産出**する成果物は **T9・T10・T12・T14** の4つだけ。それを可能にする**体感の注入**＝ **M3（Webの体）／M6（API配信・ビルド・公開）** を FOCUS（腹落ち）として支える。残りは **AI草案＋人の検品** で軽く、または **与件・参考**。

- **15タスクの“一周”は Course 0 が薄く縫う**（俯瞰・最小労力）。
- **深掘りは上記 FOCUS 群に集中**（2周目）。
- これにより「全成果物を作る重労働」を回避しつつ、全体像の俯瞰と核の習得を両立する。

```html

(function(){
  function revealAll(){document.querySelectorAll('.reveal').forEach(function(el){el.classList.add('is-in');});}

    setTimeout(revealAll,8000); /* JS不発・IO不発の保険 */

})();
</script>

```

- インタラクティブはエスケープ処理を行う（`innerHTML` 生成時の `& < >`）。
- CSS は型セレクタとクラスセレクタの相殺に注意（特にセクション間 padding/margin）。
- 日本語コメントでセクションと「編集可能領域」を明示（ブランドサイトは特に）。
- ファイル先頭にバージョン/役割コメント。
- 画像は CSS グラデーション／インライン SVG／data-URI。
- フォントは §2-7 の Google Fonts を `<link>` で（教育サイト）。ブランドサイトは §5 の方針。

## 12. リスク・落とし穴

- **単一ファイル原則の逸脱**が最大リスク。外部CSS/JS/画像に分割すると学習ループが破綻する。絶対に1枚で。
- **ブランドサイトの肥大化**：chatに載らないと改修できない。リーンに保つ（目安1500行）。
- **2つのアイデンティティの混線**：教育＝ブループリント、ブランド＝珈琲。混ぜない。
- **as-be の作り込みすぎ**：to-be を実装してしまうと生徒の課題が消える。as-is を忠実に。
- **Google Fonts 依存**：読み込み失敗時もフォールバックフォントで成立するよう、font-family にフォールバックを必ず指定。
- **対話ラボのアクセシビリティ**：ボタン化・フォーカス・aria を怠らない。
