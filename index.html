要件定義の質は、そのままプロダクトの質になる。

青写真しか描いてこなかったコンサルタントの弱点は、知識が足りないことではありません。上流で決めたことが 下流でどう跳ね返るかが見えないこと。だから要件に盲点が残り、それが「最も高くつく場所」＝実装フェーズで初めて表面化する。

この教材は、あなたをエンジニアにするのではありません。上流の判断が下流でどう効くかを、自分の手で体験させることで、その穴を塞ぎます。 そのコストを、生成AIが劇的に下げている ── それを座学ではなく、あなた自身に証明させます。

学びの中核

毎回まわす、5ステップの「体験ループ」

座学→演習ではありません。このループを1周まわすこと自体が学習です。要件が鋭ければ実装は滑らかで品質が高い。観点が抜けていれば、壁にぶつかり、手戻りし、エッジケースで壊れる。その品質差を、自分のサイトで“食らい”ます。

依頼を受ける

架空のステークホルダーから、現実的に少し曖昧な要望が届く（例：「海外の記者にプレスリリースをちゃんと見せたい」）。

AIと壁打ちして要件を定義する

ユーザーストーリー・受け入れ条件・コンテンツモデルの草案をAIに量産させる。判断（成功の定義／合意可能性／優先度）は自分が握る。

成果物をつくる

ストーリー＋受け入れ条件、コンテンツモデル、トレーサビリティ表を実際に書き起こす。

AIに実装させる

成果物をAIに渡してコードを生成し、自分の NAGI サイトに反映する。

結果を体験し、振り返る

動いたサイトを見る。そして「②の質が⑤の品質を決めた」を言語化する。

全タスク共通のフレーム

AIに「委ねる」、人が「握る」。

生成AIに量産・草案・実装を委ね、人は「成功の定義」「合意可能性」「十分性」を握る。 各モジュールでこの線引きを反復し、習慣にします。これが、AI時代のコンサルタントの新しい筋力です。

▷ AIに委ねる

  • ユーザーストーリーの量産
  • エッジケースの洗い出し
  • コンテンツモデル草案
  • 実装コードの生成
  • チェックリスト自動生成

● 人が握る

  • 成功の定義（KPI）
  • 体験原則＝設計制約
  • 合意可能性の判定
  • 優先度のトレードオフ
  • 十分性とサインオフ

学びの舞台

仮想ブランド「凪 / NAGI」

京都発、グローバル展開中の架空のプレミアム・ウェルネスブランド。多言語・多地域、ガバナンス、複数のコンテンツタイプ、ブランド表現 ── WCMの論点が“自然に必要になる”舞台です。

サイトは 構造化データ（JSON）＋クライアントサイドの描画でできた「ヘッドレスCMSのミニチュア」。コンテンツの“型”と“見た目”が分離した状態を、あなたの手で体験できます。ビルド工程ゼロ、GitHubのWebエディタで編集可能。

NAGI を訪れる →

DATA-DRIVEN

{
  "type": "product",
  "id": "nagi-essence-01",
  "name": { "ja": "凪 エッセンス", "en": "NAGI Essence" },
  "collection": "hydration",
  "key_ingredients": ["yuzu", "rice-ferment"],
  "ritual_step": 2
}

── “型”を雑に決めると、実装で壁にぶつかる。それを M5 で食らう。
