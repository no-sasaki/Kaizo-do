> 本書は、解像堂の**全モジュールと AOZORA珈琲ブランドサイトが共有する唯一の固定シナリオ**。

## 0. 物語の前提（なぜ要件定義が始まるのか）

仮想クライアント **AOZORA珈琲ホールディングス** は、グローバル・ブランドサイトを既に持っている（＝ `/brand/index.html` の v1 / as-is）。だが ──

- 新しいシングルオリジンを世界展開したいのに、**新規客が「自分に合う豆か」分からず離脱**している。
- **予約が多段で面倒**、**送料・受取が予約後にしか分からない**ため初回予約が伸びない。
- 各国でブランド表現がバラつき、統一が効いていない。

そこで「**機能追加・改修が必要になった**」。受講者（コンサルタント）は、この**既存サイトに対する要件定義→実装**を行う。これが教材全体の筋。

## 2. ブランド要求の要点（T3シード・LIGHT）

- **伝える価値**：「日常の朝を、ほんの少し上質に整える」。
- **トーン&マナー**：静かな自信、余白、誠実。過剰な煽りをしない。
- **体験原則（＝設計を縛る制約）**：
  1. 専門用語に逃げず、味を「自分ごと」に翻訳する。
  2. 迷いを最短で晴らす（不安を残さない）。
  3. 朝の数分に馴染む軽さ（速い・簡潔）。

> 人が握るのは上記**体験原則の確定**。トーンの言語化草案はAIに委ねてよい。

## 3. 優先ペルソナ（T4シード・最低限・1名のみ）

**「豆の知識が浅く、平日朝に余白がほしい30代の新規客」**

意思決定を変える属性（これだけ握る）：
- 豆の知識＝浅い → 専門用語NG、味を3軸で翻訳する必要。
- 不安の正体＝「自分に合うか分からない」→ **比較体験を最優先**。
- 来訪文脈＝平日朝の数分 → **3タップで予約完了**の導線。

（飾りの属性＝名前・年齢ちょうど・休日の過ごし方などは作らない／使わない。）

## 4. 簡易カスタマージャーニー（T7シード・最低限）

| 段階 | 顧客の問い | 感情 | サイトの仕事 |
|---|---|---|---|
| 認知 | 何のブランド？ | 無関心 | 一目で“何屋か”＋新豆の存在 |
| 興味 | どんな豆？ | 軽い好奇心 | 物語と特徴を専門用語なしで |
| **比較検討（効きどころ）** | 自分に合う？失敗しない？ | **不安・迷い（最も離脱）** | **酸味・コク・焙煎度で“自分用に翻訳”して比較** |
| 初回予約 | いくら？どう届く？面倒？ | あと一歩の慎重さ | 料金・受取を予約前に明示、3タップで完了 |
| 受取 | ちゃんと届く？淹れ方は？ | 期待＋少しの不安 | 淹れ方ガイドで初回失敗を防ぐ |
| 再訪 | また頼もうかな | 満足 or 忘却 | 好みに合わせた次の一杯を提案 |

> **効きどころ＝「比較検討」**。これが要件の重心。人が握るのはこの一点の判断。

## 5. 主要ユーザーストーリー＋受け入れ条件（T9シード・◎FOCUS）

> 受け入れ条件は「検証可能（誰が見てもDoneを判定できる）」であること。これがブランドサイト改修の実装ターゲットになる。

**US-1（Must）味で選べる**
- ストーリー：豆の知識が浅い新規客が、初めて来たとき、自分の好みに合う豆か確かめたい。なぜなら初回で失敗したくないから。
- 受け入れ条件：
  - [ ] 商品を **酸味・コク・焙煎度の3軸**で比較できる。
  - [ ] トップから「味で選ぶ／比較」へ **2タップ以内**で到達できる。
- 出どころ：ペルソナ（不安）／効きどころ（比較検討）／KPI（初回予約）。

**US-2（Must）3タップで予約**
- ストーリー：同上の客が、予約したいとき、初回予約を**3タップ以内**で完了したい。なぜなら朝の数分しかないから。
- 受け入れ条件：
  - [ ] 予約完了までの操作が **3タップ以内**。
  - [ ] 入力必須項目を最小化（初回はメール＋受取方法程度）。
- 出どころ：ペルソナ（朝の文脈）／ジャーニー（初回予約）／KPI（初回予約）。

**US-3（Must）予約前に送料・受取が分かる**
- ストーリー：同上の客が、予約を決める前に、送料と受取方法を知りたい。なぜなら不安を残したくないから。
- 受け入れ条件：
  - [ ] **送料・受取方法**が、予約に進む**前**の画面に明示される。
- 出どころ：ジャーニー（初回予約の慎重さ）／体験原則②（不安を残さない）。

## 6. コンテンツモデルの種（T10シード・◎最重要）

> 本コースの**最重要成果物**。受け入れ条件が、ここでコンテンツ型のフィールドへ落ちる。

### 6.1 主要コンテンツタイプ
商品（シングルオリジン）／ストーリー記事／店舗／採用求人／ニュース／共通（ナビ・フッタ）。

### 6.2 「商品」コンテンツ型のフィールド設計（to-be）
| フィールド | 型 | 備考 |
|---|---|---|
| 商品名 | テキスト | |
| 産地 | テキスト | 国・農園 |
| 標高 | 数値 | m |
| 精製方法 | 選択 | ウォッシュド等 |
| 説明 | リッチテキスト | 専門用語を避ける |
| 価格 | 数値 | 税込 |
| 数量限定 | 真偽 | バッジ表示 |
| **酸味** | 数値(1–5) | ★US-1の比較軸 |
| **コク** | 数値(1–5) | ★US-1の比較軸 |
| **焙煎度** | 数値(1–5) または 選択(浅/中/深) | ★US-1の比較軸 |
| テイスティングノート | テキスト | 例：ベリー、柑橘 |
| 送料区分 | 選択 | ★US-3 |
| 受取方法 | 選択(配送/店舗) | ★US-3 |
| 予約可否 | 真偽 | |

> ★印が、v1（as-is）に**無い**もの＝改修で追加するフィールド。中身（コンテンツ）と見た目（フロント）の分離（M3）が、ここで効く。

### 6.3 論理サイトマップ（簡易）
ホーム ／ 商品一覧 → 商品詳細 ／ **味で選ぶ（比較）** ／ ブランドストーリー ／ 店舗 ／ 採用 ／ ニュース。
（「味で選ぶ」が US-1 を実現する新導線。）

## 7. 方式／CMS・連携・移行の前提（T12シード・◎限定）

- **方向性を決める制約**：多言語（日英＋3言語）／既存大量コンテンツの移行／CRM・MA連携／表示速度の改善。
- **選定基準の軸（例）**：多言語運用のしやすさ／ヘッドレス対応／移行容易性／権限・ワークフロー／コスト／日本語サポート。
- **移行**：要件段階で「何を引き継ぎ、何を捨てるか」を必ず扱う（失敗の博物館：移行破綻）。
- 人が握るのは**方式の方向性・実現可能性・移行の嗅覚**。候補比較マトリクスの叩き台はAIに委ねる。全ベンダー選定はしない。

## 8. As-Is / To-Be ギャップ表（ブランドサイト v1 の作り分け）

| as-is（v1に作る） | to-be（カリキュラムで定義→生徒が実装） | 関連 |
|---|---|---|
| 商品に味の指標がない | 酸味・コク・焙煎度の3軸で比較できる | US-1／T10 |
| 予約が多段・不明瞭 | 初回予約が3タップ以内で完了 | US-2 |
| 送料・受取が予約後にしか出ない | 送料・受取を予約前に明示 | US-3 |
| 言語切替がダミー | （発展）多言語の出し分け | T12 |
| 出し分けなし | （発展）初回訪問者に“比較から”を優先表示 | 発展 |

> **エージェントへ：v1 は as-is 側を忠実に作る。to-be は作り込まない（生徒の課題）。**
> ただし★フィールドや「味で選ぶ」導線を**後から差し込みやすい構造**にし、コメントで誘導してよい。

## 9. 合意ゲートの判定例（T14・このシナリオでの「設計に渡せる」）

- US-1〜3 すべてに**受け入れ条件・優先度・出どころ**が付いている。
- 5ライン（なぜ／誰の体験／何を満たす／何の上に／どこまで）が関係者合意済み。
- 主要要件（US-1〜3）は、AI試作で**「見せて合意」済み**（M9）。
- 未決（例：多言語の出し分け範囲）は「設計で詰める前提」として明示。

→ これが揃えば、このシナリオは**設計フェーズに渡せる＝必要十分**。
